# Adaptive Sugiyama Layout Plan

Created: 2026-06-24

Status: Implemented on 2026-06-24.

Target release: `1.3.0`.

## Goal

Implement an adaptive d3-dag Sugiyama layout profile for the Revision Graph so large or wide projected graphs can trade a small amount of crossing-minimization quality for lower layout latency, while preserving the existing worker-thread layout, cache model, webview payload shape, and graph UX.

## Scope

Included:

- Add explicit layout profile selection for the existing Sugiyama path.
- Preserve the current layout as the default balanced profile.
- Add a cheaper two-layer decross profile for larger graphs.
- Keep the existing DFS decross fallback for extremely wide layers.
- Make layout cache identity sensitive to the chosen profile.
- Add trace details that make profile selection observable.
- Add focused regression tests for profile selection, cache identity, worker execution, and graph direction.

Excluded:

- No new commands, settings, menus, or visible layout mode controls.
- No switch to `grid()` or `zherebko()` in this slice.
- No webview rendering rewrite.
- No adoption of d3-dag edge route points in this slice.
- No dependency changes.

## Acceptance Criteria

- Normal graphs continue using the current balanced behavior.
- Larger graphs use the fast two-layer profile before falling back to DFS-only wide-layer decrossing.
- Extremely wide layers still choose the DFS decross path.
- Layout cache keys change when the selected profile changes.
- Cache hits, worker execution, cancellation, and fallback behavior remain intact.
- `gitRevisionGraph.traceLoading` output includes enough detail to identify the selected layout profile and graph shape.
- `npm run build` and `npm test` pass.

## Design

Introduce a small profile model in `src/revisionGraph/layout/d3DagSugiyamaLayout.ts`:

- `balanced`: current behavior, `decrossTwoLayer().order(twolayerAgg()).passes(4)`.
- `fast-two-layer`: same heuristic, lower pass count, likely `passes(1)`.
- `dfs-wide`: `decrossDfs()` for very wide layers.

Recommended first thresholds:

- `balanced` when projected node count and edge count are below large-graph thresholds.
- `fast-two-layer` when `nodes >= 800` or `edges >= 1000`.
- `dfs-wide` when any computed layer exceeds the existing `WIDE_LAYER_DECROSS_NODE_THRESHOLD` of `300`.

The exact thresholds should be constants and covered by tests. Keep them conservative because the layout already runs in a worker and the current visual quality is acceptable.

Implemented thresholds and profile behavior:

- `balanced`: selected below the large graph thresholds.
- `fast-two-layer`: selected when `nodes >= 800` or `edges >= 1000`.
- `dfs-wide`: selected when the estimated max layer width is greater than `300`.
- Balanced and fast profiles still keep the realized wide-layer DFS guard inside the d3-dag decross callback.

## Implementation Steps

1. Refactor profile selection.
   - Add `D3DagSugiyamaLayoutProfile` as a union type.
   - Add `selectD3DagSugiyamaLayoutProfile(projection)` for pre-layout graph-shape selection.
   - Keep the current graph construction unchanged.

2. Refactor decross creation.
   - Change `createRevisionGraphDecross()` to accept the selected profile.
   - `balanced` returns two-layer aggregate with 4 passes.
   - `fast-two-layer` returns two-layer aggregate with 1 pass.
   - `dfs-wide` returns DFS decross.
   - Preserve the layer-width guard so balanced/fast profiles can still route to DFS if the realized layer is too wide.

3. Return profile metadata from layout calculation.
   - Replace the raw `Map<string, D3DagSugiyamaLayoutPosition>` result with an object such as:

```ts
interface D3DagSugiyamaLayoutResult {
  readonly positions: Map<string, D3DagSugiyamaLayoutPosition>;
  readonly profile: D3DagSugiyamaLayoutProfile;
}
```

   - Update worker message shape to include `profile`.
   - Keep call sites cloning only positions where that is all they need.

4. Update cache identity.
   - Include selected profile in `buildProjectedGraphLayoutCacheKey(...)`.
   - Prefer adding the profile to the hash payload rather than making the global strategy key dynamic.
   - Bump `PROJECTED_GRAPH_LAYOUT_STRATEGY_KEY` only if persisted cache invalidation is desired for all old entries. Recommended: bump to `d3-dag-sugiyama-v3` because coordinates may change for large graphs.

5. Add trace visibility.
   - Extend `buildRevisionGraphScene(...)` or the layout boundary trace detail to include node count, edge count, selected profile, and cache hit/miss if available.
   - If changing cache stat plumbing is too invasive, include profile detail only on layout calculation paths and leave cache hit/miss as existing stats.

6. Add tests.
   - `balanced` selected for a small graph.
   - `fast-two-layer` selected for a large linear/branch graph.
   - wide-layer guard selects `dfs-wide` or routes the decross operator to DFS.
   - cache key differs between projections that cross the profile threshold.
   - worker returns positions and profile metadata.
   - existing direction assertion still passes: child/descendant rows render above visible parents.

7. Update docs/artifacts.
   - Update this plan with actual thresholds if they change during implementation.
   - Update `project-context/docs/d3-dag-performance-ux-opportunities.md` with the completed decision.
   - README only needs an update if the user-facing performance claim changes materially.

## Risks

- Thresholds may improve synthetic graphs but not real repositories; validate with at least one branch-heavy real fixture.
- Bumping the cache strategy key invalidates persisted layout positions, which is acceptable but should be called out in release notes.
- Lower decross passes can increase edge crossings in dense merge histories; keep the threshold conservative.
- Adding profile metadata through worker messages must preserve fallback to synchronous layout when the worker fails.
