# 1.4.0 Release Opening

## Goal

Open the `1.4.0` AAMAD cycle as the next minor release after the completed `1.3.0` release.

## Context

`1.3.0` is recorded as completed in the release-readiness artifact by maintainer confirmation. No `1.4.0` launch item has been selected yet, so the opening step prepares metadata and governance artifacts without claiming implementation work.

## Scope

- Bump `package.json` and `package-lock.json` to `1.4.0`.
- Create the `1.4.0` release prioritization artifact.
- Open `1.4.0` release readiness in `project-context/3.deliver/release.md`.
- Add a `1.4.0` changelog section.
- Record that implementation scope remains pending maintainer selection.

## Non-Goals

- No feature, fix, or hotfix implementation in the opening step.
- No VSIX packaging or Marketplace publication in the opening step.
- No new commands, menus, views, settings, dependencies, or contribution points.
- No broad product-surface redesign.

## Acceptance Criteria

- Version metadata is `1.4.0` in both package files.
- Release readiness clearly states that `1.4.0` is open, not packaged or published.
- The `1.4.0` scope is explicitly pending maintainer selection.
- `CHANGELOG.md` has a `1.4.0` section for the opened cycle.
- `git diff --check`, `npm run build`, and `npm test` pass after the opening updates.

## Risks

- Opening the cycle without selected launch items can make scope ambiguous. Keep `1.4.0` minor-sized until the maintainer selects specific work.
- Changelog copy should not claim fixes, features, or polish before implementation lands.
- Packaging and publication remain separate maintainer-approved actions.

## Verification

- Package metadata confirmed at `1.4.0` in `package.json`, root `package-lock.json`, and root package-lock package metadata after opening the cycle.
- `git diff --check` passed after the opening changes.
- `npm run build` passed after opening the `1.4.0` cycle.
- `npm test` passed with 459 tests after opening the `1.4.0` cycle.
