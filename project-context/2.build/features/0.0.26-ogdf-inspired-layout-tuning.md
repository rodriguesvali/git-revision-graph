# OGDF-Inspired Layout Tuning

## Goal

Reduce excessive horizontal displacement in dense revision graphs while preserving the current ELK-based layout engine and avoiding new runtime dependencies.

## Scope

- Keep `elkjs` as the layout provider.
- Tune the existing layered layout options toward the OGDF/TortoiseGit behavior observed in `SugiyamaLayout`.
- Avoid copying OGDF code or adding OGDF as a dependency because OGDF is GPL v2/v3 and this extension is MIT.

## Implementation Notes

- Explicitly use ELK `NETWORK_SIMPLEX` layering, which is the closest current ELK analogue to OGDF `OptimalRanking`.
- Keep the existing `NETWORK_SIMPLEX` node placement to limit visual risk.
- Enable `org.eclipse.elk.layered.nodePlacement.favorStraightEdges` so ELK prefers straighter vertical edge alignment over balanced horizontal placement.
- Bump the layout cache namespace from `elk-layered-v1` to `elk-layered-v2` so persisted positions from the previous heuristic are not reused.

## Acceptance Criteria

- Existing graph rendering continues to build and pass automated tests.
- Layout cache behavior remains deterministic for identical projected graph topology.
- Users with persisted `0.0.25` layout cache entries get a fresh layout under the new placement options.

## Verification

- `npm run build` passed.
- `npm test` passed with 211 tests.
- Manual validation should compare a dense branch/tag repository against the prior `0.0.25` visual layout, focusing on horizontal spread and edge readability.

## Risks

- Some graphs may trade balanced node placement for straighter edges, causing local overlap pressure or a different branch ordering.
- This is still an ELK approximation of the OGDF/TortoiseGit pipeline, not a full port of OGDF `FastHierarchyLayout`.

## Rejected Experiment

- A first-party TypeScript reconstruction of the full Sugiyama pipeline was tested after the ELK-inspired tuning was committed.
- The experiment inserted dummy nodes and minimized crossings internally, but the current webview still renders edges as direct straight paths between visible cards.
- Without carrying routed edge bends into the scene and webview renderer, dense repositories collapsed into broad same-layer card rows with heavy horizontal edge bundling.
- The experiment was reverted to the committed ELK-inspired baseline. A future full OGDF-style pass should include edge route/bend output and renderer support, not only card coordinates.

## Renderer Follow-up

- The first low-risk line-rendering experiment keeps the ELK card placement unchanged and replaces long diagonal edge paths with a single-bend path.
- Long edges now approach the target diagonally, then enter the target card through a short vertical segment so the arrow direction remains readable.
- The final vertical segment is intentionally longer than the first pass, matching the visual fan-out shape where diagonal edges close at 90 degrees near the target card.
- Edges now anchor on the side facing the target row, so upward connections leave the source card from the top and enter the target card from the bottom.
- Projected graph edges still keep child-to-parent metadata, but their path is rendered visually from parent toward descendant to match the revision graph reading direction.
- Nearly vertical or very short vertical-distance edges still render as direct lines to avoid noisy elbow artifacts.
- Rows that fan out into multiple lower-level descendants receive extra vertical clearance, and corridor pressure is counted only for edges that belong to a real fan-out group.
- Fan-out detection is based on the visual source of the edge, the parent commit/reference card, so upward fan-outs also open the crossed row corridors.
- Independent near-linear chains keep the base vertical gap so straight history columns do not become unnecessarily tall.
- Card layout now includes a row-level overlap guard so references on the same horizontal level keep the minimum horizontal gap even when the layered engine returns positions that are too close.
- The persisted scene layout key includes the final card left/width values so old manual offsets are discarded after overlap-guard changes.
- Client-side Reorganize uses left-edge spacing for same-row collision resolution, matching the server-side card overlap guard.
- This isolates edge readability from layout placement so the visual effect can be evaluated before attempting routed orthogonal paths or ELK bend-point propagation.

## Viewport Cursor Follow-up

- Empty graph viewport space uses the default cursor instead of the open-hand grab cursor.
- Drag-specific cursors remain for active viewport dragging and node grip dragging.
