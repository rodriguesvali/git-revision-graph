# Stabilization Hardening

## Define Status

Draft for the next planned release after package baseline `0.0.26`.

This artifact is not Build-approved yet. It records candidate stabilization work discovered during review.

## Goal

Reduce security and reliability risk in existing workflows without adding new user-facing feature scope.

## User Workflow

- Users continue to load the graph, inspect logs/diffs, compare refs, restore files from compare results, fetch, and run existing branch/tag workflows.
- The extension behaves more defensively around malicious or unusual repository data, very large Git output, slow Git commands, and malformed webview messages.

## Scope

- Harden targeted Git CLI argument construction for revision/ref names.
- Add output limits and timeout support to expensive Git CLI paths.
- Guard compare restore filesystem paths before writing or deleting.
- Add shared size budgets for webview messages.
- Review and, if low risk, refresh packaging dependency audit posture.

## Non-Goals

- Graph focus filters or saved filters.
- Full-history rendering.
- New Git mutation workflows.
- Marketplace publishing, version bumping, or package generation.
- Large graph architecture refactors.

## Findings

### High: option-like refs can flow into Git CLI revision arguments

- `src/revisionGraph/backend.ts:138` passes `left` and `right` directly to `git diff --no-color`.
- `src/revisionGraph/source/graphGit.ts:151` and `src/revisionGraph/source/graphGit.ts:156` build show-log revision args directly from the selected source.
- `src/revisionGraph/repository/log.ts:14` opens unified diffs through the backend using those revision strings.
- `src/refContentProvider.ts:39` passes ref strings to `repository.show`.

Impact:

- A repository-controlled ref/tag name that looks like a Git option can be interpreted as command options in some targeted Git CLI flows.

### Medium: expensive Git CLI paths lack consistent output limits/timeouts

- `src/revisionGraph/backend.ts:118` loads show-log output without a max output budget.
- `src/revisionGraph/backend.ts:139` loads unified diff output without a max output budget.
- `src/revisionGraph/backend.ts:143` loads commit details and patch output without a max output budget.
- `src/revisionGraph/controller.ts:620` runs CLI fetch with tags without timeout/cancellation.
- `src/workbenchRefActionServices.ts:204` restores file content via `git show` without an explicit output cap.

Impact:

- Large histories, huge diffs, large blobs, or slow commands can degrade the extension host or leave the UI stuck longer than expected.

### Medium: compare restore assumes Git API change paths stay inside the repository

- `src/workbenchRefActionServices.ts:199` deletes the target path from the restore plan.
- `src/workbenchRefActionServices.ts:206` creates parent directories and `src/workbenchRefActionServices.ts:207` writes restored bytes.
- The restore plan is derived in `src/compareResultRestore.ts:15` from `Change` URIs.

Impact:

- Normal Git API data should be repository-local, but the destructive filesystem invariant is implicit. Stabilization should enforce it explicitly.

### Low/Medium: webview message validators do not cap payload sizes

- `src/revisionGraph/messageValidation.ts:19`, `src/compareResults/messageValidation.ts:11`, and `src/showLog/messageValidation.ts:14` validate message shape but not maximum string or array sizes.

Impact:

- Oversized webview messages can cause avoidable extension-host work even when the action eventually does nothing.

### Low: dev-tooling audit has moderate transitive findings

- `npm audit --omit=dev --json` reports no production vulnerabilities.
- `npm audit --json` reports moderate findings through `@vscode/vsce -> @azure/identity -> @azure/msal-node -> uuid`.

Impact:

- Not a runtime extension vulnerability, but relevant to release and packaging hygiene.

## Acceptance Criteria

- Git CLI revision arguments cannot be interpreted as unintended Git options.
- Expensive Git CLI commands fail clearly when exceeding configured size or time budgets.
- Compare restore refuses to write/delete outside the repository root.
- Oversized webview messages are rejected by validators.
- Dependency audit status is recorded after any lockfile/tooling update.
- Existing compare, show log, unified diff, fetch, graph load, and restore behavior remains intact for normal repositories.

## Test Plan

- Add unit coverage for option-like tag/ref names in Git argument builders or wrappers.
- Add `gitExec` timeout tests.
- Add output-limit tests for backend diff/log/detail call sites where practical.
- Add compare restore path-containment tests.
- Add webview message validation tests for oversized strings and arrays.
- Run `npm run build`.
- Run `npm test`.
- Run `npm audit --omit=dev --json` and `npm audit --json`.

## Manual Validation

- Open the Extension Development Host with a normal Git repository.
- Verify graph loading, show log, unified diff, compare results, and compare restore workflows.
- Verify fetch with tags still works or fails with a clear message when credentials/network are unavailable.
- Validate a fixture repository containing option-like refs/tags if feasible.
- Validate behavior on a repository with a large diff or large file restore candidate.

## Release-Note Impact

Expected user-facing release notes:

- Hardened Git command handling for unusual repository ref names.
- Added safeguards for large or slow Git output paths.
- Added stricter safety checks around compare restore operations.
- Tightened webview message validation.
