# Show Log Expansion Request Cohesion

## Goal

Continue the `0.0.39` architectural cohesion work by separating Show Log commit-expansion request sequencing from the editor panel provider lifecycle.

## Scope

- Add `src/showLog/expansionRequests.ts` as the owner of Show Log commit expansion request IDs and stale-response invalidation.
- Route `ShowLogViewProvider` commit change loading through the expansion request helper.
- Invalidate pending commit expansion responses when a new Show Log source starts, when scope/filter reloads reset loaded commits, when the panel hides, and when the panel disposes.
- Preserve existing expanded commit state, cached changed-file updates, changed-file error messages, and inline change rendering.
- Add focused tests for latest-request detection and invalidation behavior.

## Acceptance Criteria

- Starting a newer commit expansion marks older expansion responses stale.
- Resetting the Show Log source, scope, filter, hidden state, or disposed panel invalidates pending expansion responses.
- Commit changed-file loads still update the same cached changes and error state when current.
- Focused Show Log expansion request, load request, shared-state, message handler, and webview tests pass.
- `npm run build` passes.
- `npm test` passes.
- `git diff --check` passes.

## Verification

- Focused Show Log expansion request, load request, shared-state, message handler, and webview tests passed with 18 tests after extracting commit expansion request tracking. This includes `npm run build` through the focused validation command.
- `npm test` passed with 335 tests after extracting Show Log commit expansion request tracking. This includes `npm run build` through the test script.
- `git diff --check` passed after extracting Show Log commit expansion request tracking and updating verification artifacts.

## Risks

- Commit expansion requests can regress if invalidation no longer lines up with source, filter, scope, hide, or panel-dispose state resets.
- The provider still owns the actual changed-file load state transitions and surrounding file/commit actions; future slices may split those if further architectural cohesion is needed.
- Runtime integration still needs manual Extension Development Host smoke testing because stale response behavior depends on live backend timing.
