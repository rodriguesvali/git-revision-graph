# 1.2.0 Indexed Graph Virtualization

## Status

Implemented for the `1.2.0` performance cycle.

## Objective

Reduce webview scroll-frame work by avoiding full node and edge collection scans every time the virtual graph viewport updates.

## Scope

- Keep the existing virtual DOM behavior: only visible/nearby graph nodes and edges are rendered into the main scene.
- Add a lightweight scene-space index for viewport candidate selection.
- Preserve existing zoom, minimap, search highlights, selection sync, context menus, persisted horizontal node offsets, retained-panel restoration, and delegated event handlers.
- Do not introduce a new rendering library or long-lived persistent graph store.

## Implementation Notes

- The webview now builds vertical bucket indexes when a ready graph scene is applied:
  - `virtualNodeIndex` stores node layouts by their vertical scene range.
  - `virtualEdgeIndex` stores graph edges by the vertical range covered by their endpoint layouts.
- Scroll and zoom frames query only the buckets intersecting the current overscanned viewport, dedupe candidates, and then apply the existing precise visibility checks.
- Node horizontal offsets remain dynamic and are still considered by the final visibility predicate, so persisted drag/offset behavior is preserved without rebuilding the vertical index.
- Edge visibility keeps the previous behavior:
  - edges connected to visible nodes remain rendered;
  - edges whose endpoint bounding range intersects the viewport can still render even if both nodes are outside the current DOM window.
- The webview shell regression test now asserts the virtual frame path uses indexed candidate collectors instead of direct `graphNodes.filter` / `graphEdges.filter` full-scene scans.

## Expected Impact

On large rendered scenes, viewport frames should inspect the nodes and edges in nearby vertical ranges rather than every loaded node and edge. The exact gain depends on graph height, branch density, long edges, current zoom, and overscan size.

## Risks

- Long edges can span many buckets, so very dense histories with many long-range edges may still produce broad edge candidate sets.
- If future layout changes introduce vertical node movement after index construction, the index must be rebuilt whenever those vertical positions change.
- Regressions would most likely appear as missing edges/nodes near viewport boundaries, especially at high zoom or after node offset restoration.

## Rollback

Restore direct `graphNodes` and `graphEdges` filtering inside `renderVirtualScene` if indexed candidates cause missing scene elements or unexpected interaction regressions.
