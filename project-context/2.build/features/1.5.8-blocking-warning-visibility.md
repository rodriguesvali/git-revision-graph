# 1.5.8 Blocking Warning Visibility

Status: Complete
Last updated: 2026-07-03
Target version: `1.5.8`

## Goal

Ensure that users cannot easily miss the reason a requested Git operation was aborted by presenting
operation-blocking precondition messages as native VS Code modal warnings.

## Baseline

- Package version opened at `1.5.8` from the unpublished `1.5.7` release-candidate baseline.
- `RefActionUi.showWarningMessage` currently maps to non-modal
  `vscode.window.showWarningMessage(message)`.
- Destructive confirmations already use modal warning messages through `RefActionUi.confirm` and
  are not part of this change.
- Packaging and Marketplace publication are out of scope until explicitly authorized.

## Scope And Acceptance Criteria

### Make Operation-Blocking Preconditions Modal

An operation-blocking precondition is a guard that terminates an action explicitly requested by the
user and uses a warning to explain why execution cannot continue.

Acceptance criteria:

- Show dirty-workspace and merge-conflict guards from `ensureWorkspaceReadyForMutation` as modal
  warnings.
- Show equivalent direct guards as modal warnings for reset, branch publication and synchronization,
  pull, push, tag operations, and stash operations.
- Show the rejected concurrent-Git-operation warning as modal when an explicitly requested mutation
  cannot start because another repository mutation is running.
- Preserve the exact abort behavior: blocked operations must not reach destructive confirmation or
  invoke a Git mutation.
- Keep existing Source Control reveal behavior for conflict guards after the warning is dismissed.
- Preserve multi-repository routing and current-repository authorization.

### Preserve Non-Blocking Message Behavior

Acceptance criteria:

- Keep destructive confirmations modal and retain their existing actions and cancellation behavior.
- Keep successful-operation feedback non-modal.
- Keep benign no-op and result-state messages non-modal, including already-current/aligned states,
  empty comparisons, and copy-to-clipboard feedback unless separately approved.
- Do not make the extension-activation warning for unavailable `vscode.git` part of this feature.

### Regression Coverage

Acceptance criteria:

- Extend the UI service contract so tests can distinguish modal blocking warnings from transient
  warnings without weakening type safety.
- Cover at least one shared dirty-workspace guard, one conflict guard, one operation-specific guard,
  and one concurrent-operation rejection.
- Assert that blocked paths do not call the underlying Git mutation.
- Preserve existing confirmation and error-message tests.

## Risks

- Excessive modal usage can interrupt routine workflows; scope is therefore limited to messages
  coupled to an aborted user-requested operation.
- Changing the asynchronous UI contract can alter ordering around Source Control reveal and refresh;
  implementations must await modal dismissal where subsequent UI actions depend on it.
- Shared test harnesses implement `RefActionUi`; the contract change may require broad but mechanical
  fixture updates.
