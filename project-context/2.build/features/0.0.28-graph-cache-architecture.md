# Graph Cache Architecture

## Build Status

Build implementation complete for the first conservative `0.0.28` cache architecture slice.

Version bump, package generation, and Marketplace publishing remain unapproved.

## Goal

Separate live repository metadata from loaded graph history enough to reuse graph snapshots across metadata-only ref changes while preserving freshness and fallback behavior.

## User Workflow

A user opens a large repository graph, then performs lightweight repository operations such as checkout, branch rename/create on an already loaded commit, or repeated refresh. The graph should keep using already loaded commit topology when safe and update visible refs/HEAD from live repository state.

## Scope

- Apply an explicit repository overlay of refs and `HEAD` before ready-state projection.
- Keep metadata-patch behavior aligned with the same overlay rules.
- Make the short-lived backend snapshot cache less sensitive to ref name and upstream metadata changes when commit topology is unchanged.
- Add trace visibility for repository overlay application.
- Add tests for overlay freshness and snapshot cache reuse across same-commit ref renames.

## Non-Goals

- Persistent commit DAG storage.
- Full projection-superset cache.
- Full-history graph rendering.
- New user-visible commands or settings.
- Replacing the existing webview/controller architecture.

## Implementation Notes

- `src/revisionGraph/panel/state.ts` now builds an explicit repository overlay from `repository.getRefs()` and `repository.state.HEAD`.
- Ready-state construction applies the overlay before projection, scene layout, primary ancestor paths, and merge-blocked target calculation.
- Metadata patch construction uses the same overlay normalization and visible-hash resolution rules as ready-state construction.
- `src/revisionGraph/backend.ts` now keys short-lived snapshot cache entries by topology-relevant ref information: ref type plus commit when available, with ref name fallback only when commit is unavailable.
- Graph load tracing now includes `state.repositoryOverlay` timing with ref count and changed/not-changed detail.

## Acceptance Criteria

- Ready graph state uses current repository refs and `HEAD` even when the cached graph snapshot contains older ref decoration names for the same commits.
- Metadata patches and ready-state builds share the same overlay normalization and visible-hash rules.
- Snapshot cache keys ignore ref names and upstream metadata when commit topology inputs are unchanged.
- Snapshot cache keys still include ref type and commit information to avoid reusing history when topology-relevant tips change.
- Existing fallback behavior remains intact when the current `HEAD` cannot be resolved inside the loaded graph.
