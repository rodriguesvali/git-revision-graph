# Show Log Load Request Cohesion

## Goal

Continue the `0.0.39` architectural cohesion work by separating Show Log log-load request sequencing and cancellation from the editor panel provider lifecycle.

## Scope

- Add `src/showLog/loadRequests.ts` as the owner of Show Log load request IDs, active `AbortController` handling, invalidation, completion, and abort-error recognition.
- Route `ShowLogViewProvider` initial load, scope reload, filter reload, load-more, hide, dispose, and panel-dispose paths through the load request helper.
- Preserve existing Show Log state transitions, load error messages, stale-response guards, load-more behavior, and panel lifecycle.
- Keep commit expansion request sequencing in the provider for now because it is tied to expanded commit state and cached changes.
- Add focused tests for current/stale request detection, active-request cancellation, invalidation, finish handling, and abort-error recognition.

## Acceptance Criteria

- Starting a newer log load marks older requests stale.
- Activating a newer log load aborts any previously active load.
- Hiding or disposing the Show Log invalidates stale responses and cancels active loads.
- Finishing an old request does not clear or abort a newer active request.
- Show Log provider behavior remains compatible with existing webview state, message dispatch, and shared-state tests.
- Focused Show Log load request, shared-state, message handler, and webview tests pass.
- `npm run build` passes.
- `npm test` passes.
- `git diff --check` passes.

## Verification

- Focused Show Log load request, shared-state, message handler, and webview tests passed with 15 tests after extracting load request tracking. This includes `npm run build` through the focused validation command.
- `npm test` passed with 332 tests after extracting Show Log load request tracking. This includes `npm run build` through the test script.
- `git diff --check` passed after extracting Show Log load request tracking and updating verification artifacts.

## Risks

- Cancellation timing can regress if active request invalidation no longer matches provider disposal or reload ordering.
- The provider still owns log state transitions and commit expansion loading; future slices may split those if further architectural cohesion is needed.
- Runtime integration still needs manual Extension Development Host smoke testing because cancellation behavior depends on live webview disposal and backend Git timing.
