# 1.0.0 Push Choice Loading Polish

## Goal

Prevent the revision graph webview from showing a blocking loading overlay before the user chooses a current-branch push mode.

## Context

The graph toolbar `Push` action opens a VS Code QuickPick so the user can choose normal push, force-with-lease, or force. Before this polish, the webview called `postMessageWithLoading(...)` as soon as the toolbar button was clicked, which blocked the graph and showed loading feedback while the user was still deciding.

## Scope

- Send the `push-current-head` message without immediate webview loading feedback.
- Keep the existing host-side push workflow, QuickPick, confirmations, refresh scheduling, and graph reload behavior unchanged.
- Update webview shell coverage so this pre-choice loading regression stays fixed.

## Non-Goals

- No change to push modes or force-push confirmation behavior.
- No change to pull, sync, fetch, reload, or other toolbar actions.
- No new command, menu, view, setting, dependency, or contribution point.

## Acceptance Criteria

- Clicking `Push` does not immediately call the blocking webview loading helper.
- The push mode QuickPick can appear without the graph entering a loading state first.
- After a push is actually performed, the existing refresh path can still show normal graph loading feedback.
- Automated coverage asserts the push toolbar action posts directly instead of using `postMessageWithLoading`.

## Risks

- Users may not see graph-local progress while the Git push command itself is running. This is preferable to blocking before user intent is confirmed, and the graph reload still provides loading feedback after a push schedules refresh.
