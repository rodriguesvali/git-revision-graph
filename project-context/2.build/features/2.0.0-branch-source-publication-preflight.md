# Branch Source Publication Preflight

Status: Complete; manual Extension Development Host validation pending
Last updated: 2026-07-16
Target version: `2.0.0`

## Goal

Prevent governed child branches from being created from an untracked local
source without first checking whether that source exists on a configured
remote. When it is unpublished, let the user publish and revalidate it before
the branch form opens.

## User Workflow

1. The user starts a release, feature, hotfix, task, or bug from a local source
   that has no configured upstream.
2. If the repository has remotes, the user selects one when necessary and the
   extension checks the same-named source branch while displaying
   `Fetching remotes...`.
3. An unpublished source offers `Publish and Continue`. An exact-sync source
   with local-only commits offers `Push and Continue`; task and bug retain the
   existing not-behind policy and may proceed from an ahead-only source.
4. A confirmed normal push establishes upstream tracking. The extension then
   revalidates the remote source and opens the branch form only when the result
   is synchronized.
5. Cancellation, read-only remotes, remote-ahead/diverged/inconclusive state,
   push failure, or failed revalidation leaves the form closed.

Repositories with no configured remotes retain the existing local-only branch
creation workflow.

## Scope And Non-goals

- Reuse the Pull Request source publication classifier and existing blocking
  remote-fetch loading lifecycle.
- Keep all work inside the existing guarded repository mutation.
- Never force-push, create the requested child branch before preflight success,
  or automatically reconcile remote-ahead/diverged sources.
- Do not change the established tracked-upstream synchronization path. A stale
  configured upstream whose remote branch was deleted still fails its explicit
  fetch and requires manual repair.
- No protocol, command, setting, dependency, version, package, or publication
  changes.

## Acceptance Criteria

- A source without upstream is checked when at least one remote exists.
- Missing remote source branches are published only after explicit confirmation
  and use upstream tracking.
- The form opens only after post-push revalidation returns `ready`.
- Declined publication, read-only remote, failed push, and failed revalidation
  block the form with no child branch creation.
- No-remotes repositories remain able to create local-only governed branches.
- All warnings and errors use modal presentation; loading clears before prompts
  and failure dialogs.

## Verification And Manual Validation

- Automated coverage includes successful publish/revalidate, declined
  publication, no-remotes fallback, read-only remote, and failed revalidation.
- Required gates: `npm run quality:check`, `npm run build`, `npm test`,
  `npm run test:platform`, `git diff --check`, and `graphify update .`.
- Manual smoke: create an unpublished feature, run `Start New Task`, confirm
  `Fetching remotes...`, accept `Publish and Continue`, and verify the feature
  gains upstream tracking before the task form opens. Repeat with cancellation,
  a read-only remote, and no configured remote.
