# 1.2.0 Incremental Show Log Updates

## Status

Implemented for the `1.2.0` performance cycle.

## Objective

Reduce Show Log host-to-webview payload size and DOM churn for repeated paging by applying `loadMore` results incrementally instead of rebuilding and resending the complete loaded commit list after each page.

## Scope

- Keep full-state messages for initial render, retained-panel recovery, source changes, filter changes, scope changes, commit expansion, and error recovery.
- Add an incremental append message for successful `loadMore` requests.
- Preserve source-token cancellation, stale patch guards, commit comparison selection, context menus, inline file actions, filtering, retained panel state, and Compare Results routing.
- Avoid introducing a new persistence layer or changing the Show Log product surface.

## Implementation Notes

- `ShowLogViewProvider.loadMore` still posts a full state when paging begins so the existing loading indicator path is preserved.
- On successful page load, the provider now posts an `append` message containing:
  - the active `sourceToken`;
  - the expected previous commit count;
  - the new summary count and `hasMore` state;
  - only the newly appended webview commit items.
- The webview applies the patch only when the source token and previous commit count match the current state.
- If a patch is stale or cannot be aligned, the webview asks the host for the full state again through the existing `ready` recovery path.
- The webview appends new commit DOM entries and updates only the summary/load-more block for the successful append path, instead of rerendering the whole content body.
- Commit lane topology for appended commits is still built through the existing Show Log view-state mapper, preserving graph-line correctness.

## Expected Impact

Repeated Show Log paging now sends and renders the new page instead of repeatedly sending and rebuilding all already loaded commits. The impact grows with the number of loaded pages in one retained Show Log panel.

## Manual Validation Focus

- Open Show Log for a branch or range with more than one page of commits.
- Scroll until `loadMore` triggers repeatedly.
- Confirm appended commits appear in order and graph lanes remain visually coherent.
- Confirm filtering, Show All Branches, commit expansion, two-commit compare selection, context menus, file actions, and retained-panel restoration still behave normally.

## Risks

- Incremental messages add a second host-to-webview state path; stale patch guards must continue to fall back to full-state recovery.
- Future changes that make already-rendered commit topology depend on later pages may need either broader patch data or a full-state rerender after append.

## Rollback

Restore `loadMore` success to call `postState()` and remove the `append` webview message handler if incremental append behavior regresses Show Log rendering or interactions.
