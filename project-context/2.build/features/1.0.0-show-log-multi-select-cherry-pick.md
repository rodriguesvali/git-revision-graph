# Show Log Multi-Select Cherry Pick

## Goal

Improve the Show Log developer workflow by allowing more than two commit selections and exposing a `Cherry Pick` action from the commit context menu.

## Scope

- Allow `Ctrl` + left-click selection to keep more than two commits selected in Show Log.
- Clear the current selection on any normal left click over a commit row so users can leave multi-selection mode without a separate clear command.
- Drive selected-row emphasis from `data-selected` so the first selected commit and every commit above the two-item compare selection remain visibly selected.
- Keep the `Base` badge constrained to the two-item comparison state.
- Add a `Cherry Pick` context action:
  - If the context-clicked commit is selected, cherry-pick the selected commits.
  - If the context-clicked commit is not selected, cherry-pick only that commit.
- When the context-clicked commit is part of a multi-selection, show only `Cherry Pick` in the context menu.
- Run cherry-pick through targeted Git CLI because the public VS Code Git API does not expose this operation.
- Preserve existing workspace mutation guards before executing the operation.
- Detect an existing `CHERRY_PICK_HEAD` before starting a new cherry-pick.
- Offer a guarded `Abort Cherry Pick` fallback when a previous cherry-pick is still in progress.
- Detect Git's empty cherry-pick output after partial sequence application.
- Offer `Skip Empty Commit` before offering the destructive abort fallback for empty cherry-picks.
- Open Source Control if the user keeps the pending cherry-pick.

## Acceptance Criteria

- Multi-selection above two commits persists in the Show Log webview state.
- A normal left click on any commit row clears the active selection before the row expansion action continues.
- With exactly two commits selected, the single `Base` badge remains available, but the context menu offers only `Cherry Pick`.
- With more than two commits selected, context menus offer only `Cherry Pick` and no `Base` badge is rendered.
- `Cherry Pick` appears in commit context menus and sends the selected commit list to the extension host.
- Cherry-pick is blocked when the workspace is dirty or conflicted.
- When a cherry-pick is already in progress, a new cherry-pick is not started.
- When Git stops on an empty cherry-pick, the extension offers `Skip Empty Commit` before showing generic failure feedback.
- If the user confirms `Skip Empty Commit`, the extension runs `git cherry-pick --skip`, refreshes the graph, and reports the skip.
- If the user confirms `Abort Cherry Pick`, the extension runs `git cherry-pick --abort`, refreshes the graph, and stops there.
- If the user does not confirm the abort, Source Control opens for manual resolution.
- Successful cherry-pick refreshes the graph and reports concise feedback.
- Failed cherry-pick reports the Git error, opens Source Control, and schedules a graph refresh for conflict visibility.
