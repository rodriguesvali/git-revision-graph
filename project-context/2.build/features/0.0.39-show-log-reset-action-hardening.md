# 0.0.39 Show Log Reset Action Hardening

## Goal

Continue the architectural bad-practices hardening work by moving Show Log reset-to-commit action policy out of the editor panel provider.

## Problem

`ShowLogViewProvider` still owned reset-to-commit policy: locating the loaded commit entry, handling unavailable Git action services, deriving the commit label, and invoking `resetCurrentBranchToCommit`. That mixed panel lifecycle and state visibility checks with branch mutation workflow setup.

## Scope

- Added `src/showLog/resetAction.ts` as the owner for Show Log reset-to-commit action setup.
- Routed `ShowLogViewProvider` through `resetShowLogCommit` after visible-state and repository validation.
- Preserved existing behavior for:
  - ignoring unloaded commits;
  - showing `Could not reset the branch because Git actions are not ready yet.` when Git actions are unavailable;
  - preferring `shortHash` and falling back to the first 8 characters of the full hash;
  - calling the existing `resetCurrentBranchToCommit` workflow.
- Added focused tests for commit labels, unloaded commits, unavailable services, and workflow invocation.

## Acceptance Criteria

- `ShowLogViewProvider` no longer owns reset-to-commit workflow setup.
- Reset action behavior is testable without loading VS Code extension-host APIs.
- Existing Show Log message dispatch, loaded-state lookup, reset workflow, and rendering behavior remain unchanged.
- Focused Show Log validation and full automated verification pass.

## Risks

- Manual Extension Development Host smoke testing should still confirm reset-to-commit prompts and refresh behavior from Show Log.
