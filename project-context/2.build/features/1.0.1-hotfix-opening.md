# 1.0.1 Hotfix Opening

## Goal

Open the `1.0.1` cycle as a narrow hotfix release after the published `1.0.0` release.

## Context

`1.0.0` is published and remains the latest Marketplace release. A force-push authentication bug was reported in the current-branch `Push` workflow, and investigation confirmed the hotfix should route force modes through the built-in VS Code Git API force parameter instead of direct non-interactive Git CLI execution.

## Scope

- Bump `package.json` and `package-lock.json` to `1.0.1`.
- Create the `1.0.1` release prioritization artifact.
- Open `1.0.1` release readiness in `project-context/3.deliver/release.md`.
- Add a `1.0.1` changelog section.
- Register the force-push Git API hotfix as the first launch item.

## Non-Goals

- No hotfix implementation in the opening step.
- No VSIX packaging or Marketplace publication in the opening step.
- No new commands, menus, views, settings, dependencies, or contribution points.
- No broad push workflow redesign.

## Acceptance Criteria

- Version metadata is `1.0.1` in both package files.
- Release readiness clearly states that `1.0.1` is open, not packaged or published.
- The force-push bug is listed as the first planned fix.
- `CHANGELOG.md` has a `1.0.1` section for the hotfix cycle.
- `git diff --check` and `npm run build` pass after the opening updates.

## Risks

- The hotfix cycle can drift into broader push workflow changes. Keep the release limited to the force-push authentication path unless the maintainer explicitly approves more.
- Changelog copy should not claim the fix is implemented until the code change lands.
- Packaging and publication remain separate maintainer-approved actions.

## Verification

- Package metadata confirmed at `1.0.1` in `package.json`, root `package-lock.json`, and root `package-lock.json` package metadata after opening the cycle.
- `git diff --check` passed after opening artifacts.
- `npm run build` passed after opening the `1.0.1` cycle.
