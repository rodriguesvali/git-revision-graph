# 2.0.0 Sync Promotion PR Context

Status: Implemented; manual Extension Development Host validation pending
Updated: 2026-07-16

## Goal

Keep each equalization `sync/*` branch tied to the branch from which the action was requested and,
after the equalization merge is complete, let the user open `Promotion PR Context` back to that
exact branch.

## User Workflow

1. The user requests `Prepare Equalization` from a release or feature target and selects an origin.
2. The extension creates and checks out `sync/*` from the requested target, records that target in
   local branch Git configuration, and merges the selected origin into `sync/*`.
3. If Git leaves a merge in progress, the user resolves conflicts and completes the merge in Source
   Control. Pull Request context remains blocked until `MERGE_HEAD` is cleared.
4. `Flow Governance > Promotion PR Context` on `sync/*` uses only the recorded release or feature
   target, fetches its remote tip, and performs the normal target/source preflights.

## Scope

- Persist `branch.<sync>.git-revision-graph-flow-target` before equalization merge.
- Enrich Flow Governance sync metadata from repository-local Git configuration on graph load.
- Generate one Pull Request target per mapped sync branch; do not pair sync branches with every
  active release.
- Govern both `sync -> release` and `sync -> feature` transitions.
- Authorize sync handoff only for the mapped target.
- Block sync handoff with a modal warning while a merge is in progress.

## Non-goals

- Automatic conflict resolution, merge commit creation, push, PR creation, or sync branch cleanup.
- Inferring targets for legacy sync branches that do not have persisted metadata.
- Adding provider APIs, settings, dependencies, commands, or version changes.

## Acceptance Criteria

- The sync branch is based on the action target and merges the selected origin.
- Target metadata is durable across refresh and extension restart, including conflict resolution.
- Failure to persist required target metadata prevents the merge from starting.
- A sync branch exposes one PR handoff to its recorded release or feature target.
- Resolved files alone do not unlock handoff; the merge must be committed or aborted.
- Host authorization rejects any alternate target supplied by the webview.
- Existing feature target selection and production promotion behavior remain unchanged.

## Manual Validation

- Prepare equalization from a feature, force a merge conflict, resolve the files, and verify PR
  context remains blocked until the merge commit is completed.
- Refresh or reopen the graph, then verify the sync submenu targets the original feature.
- Repeat from a release and verify the sync submenu targets that release.
- Verify no push occurs during equalization and normal publish/push confirmation still occurs in
  the PR source preflight.

## Release-Note Impact

Document that equalization sync branches retain their original target and provide guarded PR
handoff after merge completion.
