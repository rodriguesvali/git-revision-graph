# 0.0.24 Graph Layout Layering

## Goal
Improve the revision graph organization so visible commits are placed by graph layers instead of a strict linear `git log` row sequence.

## User Workflow
Users compare the VS Code graph with TortoiseGit on the same repository and expect related branches, merge parents, and independent chains to appear in coherent horizontal layers rather than as one tall narrow column.

## Scope
- Preserve the existing ELK layered layout dependency.
- Keep the current webview, controller, projection, minimap, and interaction model.
- Use both horizontal and vertical coordinates from ELK when building the scene.
- Compact ELK vertical coordinates into stable graph rows for the existing webview renderer.
- Add regression coverage for layered rows.

## Non-Goals
- Full TortoiseGit parity.
- Custom edge routing or bend point rendering.
- Replacing the projection model or Git snapshot loading.
- Changing graph colors, menus, minimap behavior, or toolbar commands.

## Observations
- TortoiseGit spreads related subgraphs horizontally and allows nodes at the same graph depth to share a row.
- The current product calculates horizontal placement with ELK but discards ELK vertical placement.
- Because rows are assigned from projection order, the graph becomes very tall and narrow, producing long vertical or diagonal edges and a needle-like minimap.

## Acceptance Criteria
- The scene builder preserves ELK `x` and `y` positions.
- Commits at the same graph layer can share the same webview row.
- Merge parents remain horizontally distinct.
- Existing metadata patch and viewport preservation continue to use `sceneLayoutKey`.
- Automated tests cover the row-compaction behavior.

## Manual Validation
- Open the same repository shown in the screenshots in the Extension Development Host.
- Use `All Refs`.
- Compare the initial render and `Reorganize` result against TortoiseGit:
  - graph should be less needle-shaped in the minimap;
  - independent chains should share horizontal layers where possible;
  - merge parent cards should remain separated laterally;
  - no card overlap after auto-arrange.

## Release-Note Impact
- Improved revision graph layout by using ELK layered rows, reducing overly tall/narrow graph organization in branch-heavy repositories.

## Handoff Notes
- This is an incremental layout improvement. Edge routing still uses straight lines, so long-crossing edges can remain in complex histories. A future cycle can preserve ELK edge sections and render routed paths.
