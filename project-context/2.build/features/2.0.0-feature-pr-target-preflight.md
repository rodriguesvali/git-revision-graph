# Feature Pull Request Target Preflight

Status: Complete; manual Extension Development Host validation pending
Last updated: 2026-07-16
Target version: `2.0.0`

## Goal

Prevent `feature -> release` Pull Request handoff from relying on a stale local
release. The trusted host workflow must fetch the selected remote release,
require the local release to match it exactly, and only then check whether the
feature has commits ahead.

## User Workflow

1. The feature context form always opens without a selected target and lists
   every classified release, even when only one release is available. The user
   must explicitly choose the target; opening the form alone performs no fetch.
   After selection, cached local target status remains immediate feedback, but
   the choice is sent to the host and cannot bypass the remote preflight.
2. Before populating trusted Pull Request context, the host resolves the handoff
   remote, reads and fetches the selected release tip, and compares it with the
   local release.
   The existing loading overlay displays `Fetching remotes...` while this remote
   work is active.
3. A synchronized release proceeds to the existing ahead check and source
   publication preflight.
4. A local release that is behind, ahead, divergent, missing remotely, or
   inconclusive produces a modal warning and blocks context/URL handoff.
5. The user synchronizes the release explicitly, refreshes the graph, and
   retries.

## Scope And Non-goals

- Reuse the existing bounded remote-tip loader, mutation coordinator, and
  target-comparison logic used by production promotion.
- Apply the gate to both context generation and provider URL opening, which
  already rerun the trusted host preflight.
- Preserve the feature form and its multi-release target selector.
- Do not pull, merge, checkout, create an equalization branch, update a local
  branch, push a target, or force-push. Fetch may update remote-tracking data
  only.
- Do not change branch classification, provider adapters, webview protocol,
  commands, settings, dependencies, version, packaging, or publication.

## Acceptance Criteria

- `feature -> release` loads the selected release from the resolved remote
  before the trusted ahead check.
- The feature form never auto-selects a release. Its context actions remain
  disabled and no target fetch starts until the user explicitly selects one.
- The ahead check receives the fetched remote commit and requires exact
  local/remote target synchronization.
- Behind, ahead, and divergent local releases all fail closed with an awaited
  modal warning; source publication and context generation do not run.
- A synchronized release proceeds in this order: target remote verification,
  target ahead check, source publication verification, context/URL handoff.
- Existing `release/hotfix -> main` synchronization and ancestry behavior is
  preserved.
- Fetching a target performs no pull, merge, branch creation, target push, or
  equalization.

## Risks And Controls

- **Network failure or missing release:** fail closed with no handoff.
- **Concurrent repository operation:** target fetch remains protected by the
  per-repository mutation coordinator.
- **Stale webview state:** the host reruns target and source checks immediately
  before each trusted handoff.
- **Unexpected local mutation:** tests assert that target loading invokes only
  fetch and leaves pull, merge, branch creation, and push untouched.

## Release-note Impact

Flow Governance now requires an explicit feature target release selection and
then verifies that its local ref matches the freshly fetched remote tip before
allowing Pull Request handoff.
