# 1.0.2 Release Opening

## Goal

Open the `1.0.2` cycle as the next patch release after the completed `1.0.1` release.

## Context

`1.0.1` is recorded as completed in the release-readiness artifact. No `1.0.2` launch item has been approved yet, so the opening step prepares metadata and governance artifacts without claiming implementation work.

## Scope

- Bump `package.json` and `package-lock.json` to `1.0.2`.
- Create the `1.0.2` release prioritization artifact.
- Open `1.0.2` release readiness in `project-context/3.deliver/release.md`.
- Add a `1.0.2` changelog section.
- Record that implementation scope remains pending maintainer selection.

## Non-Goals

- No feature or hotfix implementation in the opening step.
- No VSIX packaging or Marketplace publication in the opening step.
- No new commands, menus, views, settings, dependencies, or contribution points.
- No broad product-surface redesign.

## Acceptance Criteria

- Version metadata is `1.0.2` in both package files.
- Release readiness clearly states that `1.0.2` is open, not packaged or published.
- The `1.0.2` scope is explicitly pending maintainer selection.
- `CHANGELOG.md` has a `1.0.2` section for the opened cycle.
- `git diff --check` and `npm run build` pass after the opening updates.

## Risks

- Opening the cycle without selected launch items can make scope ambiguous. Keep `1.0.2` patch-sized until the maintainer selects specific work.
- Changelog copy should not claim fixes or polish before implementation lands.
- Packaging and publication remain separate maintainer-approved actions.

## Verification

- Package metadata confirmed at `1.0.2` in `package.json`, root `package-lock.json`, and root package-lock package metadata after opening the cycle.
- `git diff --check` passed after opening artifacts.
- `npm run build` passed after opening the `1.0.2` cycle.
