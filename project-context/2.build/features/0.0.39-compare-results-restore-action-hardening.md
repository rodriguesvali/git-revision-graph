# 0.0.39 Compare Results Restore Action Hardening

## Goal

Continue the architectural bad-practices hardening work by moving Compare Results restore confirmation and execution policy out of the editor panel provider.

## Problem

`CompareResultsViewProvider` still owned the destructive restore workflow for a changed file: deciding whether the item can be restored, building the confirmation prompt, invoking the restore service, and translating restore failures. That mixed panel lifecycle/state refresh with mutation policy and made the confirmation/error path harder to test directly.

## Scope

- Added `src/compareResults/restoreAction.ts` as the owner for Compare Results worktree restore confirmation and execution.
- Routed provider restore actions through `restoreCompareResultItemToWorktree`.
- Kept `CompareResultsViewProvider` responsible for refreshing the worktree comparison only after a restore succeeds.
- Preserved existing behavior for:
  - ignoring non-worktree items;
  - using `worktreeLabel` in confirmation when available;
  - using `worktreeRef` as the restore ref;
  - reporting restore failures through the existing operation error message.
- Added focused tests for confirmation text, skipped restores, successful restores, cancellation, and failure reporting.

## Acceptance Criteria

- `CompareResultsViewProvider` no longer owns restore confirmation or restore error formatting.
- Restore behavior remains testable without loading VS Code extension-host APIs.
- Existing Compare Results item actions, message dispatch, view-state, and rendering behavior remain unchanged.
- Focused Compare Results validation and full automated verification pass.

## Verification

- `npm run clean:test && npm run build && tsc -p ./tsconfig.test.json && node --test out-test/test/compareResultsRestoreAction.test.js out-test/test/compareResultsItemActions.test.js out-test/test/compareResultsViewState.test.js out-test/test/compareResultsMessageHandler.test.js out-test/test/compareResultsWebview.test.js` passed with 21 tests.

## Risks

- Restore behavior depends on preserving the provider's previous refresh-after-success behavior.
- The default service adapter lazily loads VS Code and workbench restore services so pure tests do not require extension-host APIs.
