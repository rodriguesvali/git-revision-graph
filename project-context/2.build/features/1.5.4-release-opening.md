# 1.5.4 Release Opening

## Goal

Open the `1.5.4` cycle as a narrow UX safety patch after the published `1.5.3` release.

## Context

`1.5.3` is recorded as published in the release-readiness artifact. A follow-up review found
remaining blocking-alert cases that were still surfacing as standard VS Code notifications,
including the current-branch sync path when the remote server rejects the push side for missing
write privileges.

## Scope

- Open the `1.5.4` release readiness state in `project-context/3.deliver/release.md`.
- Create the `1.5.4` release prioritization artifact.
- Add a `1.5.4` changelog candidate section.
- Include the remote permission modal-alert correction as the first implemented scope item.
- Apply the package metadata bump to `1.5.4` after explicit maintainer approval.

## Non-Goals

- No VSIX packaging or Marketplace publication in the opening step.
- No command, menu, view, setting, dependency, or contribution-point change.
- No broad rewrite of Git error handling or notification policy.

## Acceptance Criteria

- Release readiness clearly states that `1.5.4` is open and not packaged or published.
- The remote permission modal-alert fix is included in the `1.5.4` scope.
- Existing `1.5.3` publication records remain intact.
- Package metadata reports `1.5.4` after version-bump approval.

## Risks

- Over-classifying generic Git failures as blocking could regress the nonblocking sync/pull
  loading behavior restored in `1.2.0`.
- Under-classifying protected-branch or server authorization failures could leave release-impacting
  errors in transient toasters.

## Verification

- Package metadata reports `1.5.4` in `package.json`, root `package-lock.json`, and root
  package-lock package metadata after maintainer approval.
- The included modal-alert fix passed `npm run build`, focused compiled tests for
  `errorDetail` and `refActions`, full `npm test` with 508 tests, and `git diff --check`.
