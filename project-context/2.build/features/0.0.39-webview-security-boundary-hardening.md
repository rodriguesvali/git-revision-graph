# 0.0.39 Webview Security Boundary Hardening

## Goal

Continue the architectural bad-practices hardening work by centralizing webview nonce and CSP construction outside the revision graph renderer internals.

## Problem

`Compare Results` and `Show Log` imported `createNonce` from `src/revisionGraph/webview/shared.ts`. That coupled review-panel webviews to graph-renderer helpers for a host security concern, and each webview duplicated the same Content Security Policy literal.

## Scope

- Added `src/webviewSecurity.ts` as the neutral owner for webview nonce creation and CSP construction.
- Moved nonce creation out of `revisionGraph/webview/shared.ts`.
- Routed the revision graph, Compare Results, and Show Log HTML renderers through the shared webview security helper.
- Preserved the existing CSP behavior: `default-src 'none'`, inline styles, and nonce-scoped scripts.
- Moved nonce coverage into focused webview security tests and added CSP string coverage.

## Acceptance Criteria

- Review-panel webviews no longer import security helpers from revision graph renderer internals.
- All shipped webview surfaces use the same CSP builder.
- Existing webview HTML and runtime behavior remain unchanged.
- Focused webview/security validation and full automated verification pass.

## Verification

- `npm run clean:test && npm run build && tsc -p ./tsconfig.test.json && node --test out-test/test/webviewSecurity.test.js out-test/test/revisionGraphWebviewShared.test.js out-test/test/revisionGraphWebview.test.js out-test/test/compareResultsWebview.test.js out-test/test/showLogWebview.test.js` passed with 35 tests.

## Risks

- If a future webview needs a different CSP, it should extend the central security helper instead of reintroducing per-view literals.
- Manual Extension Development Host smoke testing should still confirm all three webview surfaces render and execute scripts normally.
