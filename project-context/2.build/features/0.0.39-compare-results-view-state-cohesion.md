# Compare Results View State Cohesion

## Goal

Continue the `0.0.39` architectural cohesion work by separating Compare Results webview state mapping from the editor panel provider lifecycle.

## Scope

- Add `src/compareResults/viewState.ts` as the owner of Compare Results state-to-webview mapping.
- Move Compare Results item collection and item-to-webview projection out of `CompareResultsViewProvider`.
- Keep the provider responsible for panel lifecycle, message callbacks, diff actions, clipboard actions, restore confirmation, and worktree refresh.
- Preserve existing Compare Results summaries, labels, item IDs, sorting, rename display, and webview payload shape.
- Add focused tests for empty, between-ref, worktree, rename, and item lookup state mapping.

## Acceptance Criteria

- Empty Compare Results state maps to the current empty webview message.
- Between-ref Compare Results state keeps the current source/target labels and sorted items.
- Worktree Compare Results state keeps `Worktree` as target label and preserves worktree ref metadata.
- Renamed file entries keep the current original path, display path, basename, directory, and full path mapping.
- Provider item lookup continues to use the same generated item IDs.
- Focused Compare Results view-state, shared-state, webview, and message handler tests pass.
- `npm run build` passes.
- `npm test` passes.
- `git diff --check` passes.

## Verification

- Focused Compare Results view-state, shared-state, webview, and message handler tests passed with 13 tests after extracting webview state mapping. This includes `npm run build` through the focused validation command.
- `npm test` passed with 327 tests after extracting Compare Results webview state mapping. This includes `npm run build` through the test script.
- `git diff --check` passed after extracting Compare Results webview state mapping and updating verification artifacts.

## Risks

- Webview payload compatibility can drift if state mapping changes labels, item IDs, or rename fields.
- Provider item lookup depends on the same item projection used by webview state; tests now cover the shared helper, but live webview behavior still needs manual Extension Development Host smoke testing.
