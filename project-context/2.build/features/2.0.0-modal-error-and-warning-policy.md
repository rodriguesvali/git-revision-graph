# 2.0.0 Modal Error And Warning Policy

Status: Complete
Last updated: 2026-07-16
Target version: `2.0.0`

## Goal

Apply the maintainer-approved notification policy: every error is presented as a native modal
dialog, and a warning that allows its workflow to continue is also presented modally.

## Scope

- Centralize VS Code error presentation in one modal workbench presenter.
- Route direct Revision Graph, Compare Results, Show Log, document, and ref-action errors through
  that presenter.
- Post current Flow Governance state before every error dialog so loading feedback clears before
  the modal opens.
- Make description-persistence warnings modal while allowing equalization and branch creation to
  continue after dismissal.
- Preserve information toasts and warning paths that terminate the current flow.

## Non-goals

- No message-copy rewrite, new commands, settings, dependencies, contribution points, version bump,
  packaging, or publication.
- No conversion of information messages or flow-terminating warning toasts in this slice.

## Acceptance Criteria

- Production TypeScript has no direct `vscode.window.showErrorMessage` use outside the shared
  modal presenter.
- The shared presenter forces `{ modal: true }` while preserving optional detail text.
- Flow Governance posts current state before every error and does not leave loading feedback behind
  the dialog.
- Equalization and flow-branch description persistence warnings are modal, and their workflows
  continue after dismissal.
- Existing Git guards, conflict details, Source Control handoff, refresh, cancellation, and
  multi-repository behavior remain unchanged.

## Risks And Verification

- Awaited error paths now wait for dialog dismissal; intentionally detached sync/pull error promises
  retain their existing state-clearing behavior, with the Flow wrapper posting state first.
- Run focused notification, Flow Governance, ref-action, Show Log, Compare Results, and repository
  document tests, followed by `npm run quality:check`, `npm run build`, `npm test`,
  `git diff --check`, and `graphify update .`.
- Manual smoke remains required for generic Git failures, conflict detail dialogs, description-save
  warning continuation, and loading-state clearance before modal errors.

## External Contract

- VS Code `MessageOptions` supports modal messages on the pinned `@types/vscode` `1.90.0` baseline.
- The returned thenable resolves after action selection or dismissal; continued warning workflows
  therefore resume after the user closes the dialog.

## Handoff

- No Marketplace metadata impact beyond the user-visible notification behavior recorded in the
  changelog and release readiness notes.
- Implementation routes direct and ref-action error presentation through `showModalErrorMessage`,
  keeps VS Code loading lazy for injected test workflows, and adds a quality gate against direct
  error-message API usage outside the shared presenter.
- `npm run quality:check` passed with 216 production files and 2,059 functions; `npm test` passed
  all 725 tests on 2026-07-16. Manual Extension Development Host smoke remains pending.
