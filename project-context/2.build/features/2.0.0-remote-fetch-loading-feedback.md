# Remote Fetch Loading Feedback

Status: Complete; manual Extension Development Host validation pending
Last updated: 2026-07-16
Target version: `2.0.0`

## Goal

Make automatic remote fetches visible in the Revision Graph instead of leaving
the user with an apparently unresponsive Flow Governance action.

## User Workflow

1. The user starts a governed branch preflight or explicitly selects the target
   release for a feature Pull Request handoff that needs current remote
   positions. Merely opening the feature form does not fetch.
   A cached local `not-ahead` or inconclusive feature-target status no longer
   short-circuits the trusted host request.
2. While the extension probes/fetches the remote, the existing blocking loading
   overlay displays `Fetching remotes...` with its spinner and live status.
3. The overlay clears as soon as that remote operation settles.
4. Eligibility checks, native confirmations, warnings, errors, forms, and URL
   handoff continue only after the overlay has cleared.

The explicit toolbar fetch already used this same loading presentation and
remains unchanged.

## Scope

- Add a shared Flow Governance wrapper around automatic remote operations.
- Cover tracked-upstream fetches before governed branch creation.
- Cover publication checks and post-push revalidation for untracked governed
  branch sources when a repository remote is configured.
- Cover the confirmed non-current fast-forward fetch used by branch-start
  synchronization.
- Cover Pull Request target verification, source publication verification, and
  post-push source revalidation.
- Keep cached feature-target feedback visible in the multi-release form while
  requiring every explicitly selected release to run the trusted host preflight.
- Reuse the existing `set-loading` host message, loading overlay, spinner,
  accessibility behavior, and `postCurrentState` clearing path.

## Non-goals

- No new dialog markup, protocol message, command, setting, dependency, version
  bump, packaging, or publication.
- No loading overlay while waiting for Quick Pick, confirmation, warning, or
  error dismissal.
- No change to fetch options, target/source eligibility, synchronization,
  publication, mutation coordination, or failure policy.

## Acceptance Criteria

- Every automatic `repository.fetch(...)` in Flow Governance runs inside the
  shared loading lifecycle.
- Pull Request remote probes that may include `ls-remote` plus fetch use the
  same lifecycle.
- Every selected feature release reaches the host preflight even when cached
  local state says `not-ahead`, `unknown`, or otherwise blocked.
- Opening the feature context form never auto-selects its sole release and does
  not start remote work; selection is the fetch trigger.
- The label is exactly `Fetching remotes...`, matching the explicit toolbar
  fetch.
- Success and failure both clear the overlay through `finally`.
- The overlay is already cleared before a confirmation, warning, error, form,
  or provider URL is presented.
- Existing Git safety and multi-repository coordination remain unchanged.

## Risks And Controls

- **Stuck overlay after failure:** the shared wrapper clears loading in
  `finally`, with explicit success/failure coverage.
- **Overlay behind native modal UI:** the wrapper ends before the workflow
  evaluates and presents confirmations or failures.
- **Over-broad busy state:** only the remote operation is wrapped; local checks
  and user decision time are not.
