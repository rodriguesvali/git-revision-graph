# Graph Layout Stabilization

## Build Status

Build implementation complete for the `0.0.27` graph layout stabilization slice.

This artifact records the investigation and fix for divergent graph disposition between normal refresh/update paths and the full rebuild observed after fetch. Version bump, package generation, and Marketplace publishing remain unapproved.

## Goal

Keep the revision graph disposition deterministic for the same projected topology and prevent stale browser-side node offsets from masking the latest default layout.

## User Workflow

- Users open a large repository graph and inspect a fan-out area around the current branch.
- Users can switch branches, refresh, fetch, or let the extension run "Updating revision graph".
- The graph should settle into the same balanced disposition for the same graph topology, instead of preserving an older skewed arrangement.

## Scope

- Investigate why screenshots of the same graph region can differ before and after a fetch-triggered update.
- Make scene layout identity sensitive to the rendered layout algorithm version and edge topology.
- Preserve the existing webview state model, persisted offsets, and manual node dragging behavior when the topology and layout algorithm are unchanged.
- Add regression coverage for stale-offset invalidation when edge topology changes.

## Non-Goals

- Replace the current ELK/projection layout pipeline.
- Add a new persistent graph layout store.
- Change branch filtering, fetch behavior, or repository snapshot loading.
- Remove manual node dragging or stored offsets.

## Findings

- The better screenshot after fetch is consistent with a full graph rebuild using the newest fan-out balancing behavior.
- Earlier screenshots can plausibly preserve browser-side `nodeOffsets` from a previous scene when the layout identity is considered unchanged.
- The previous `sceneLayoutKey` was based only on node hash, row, rounded default left, and width.
- That key did not include a layout-rule version or rendered edge topology, so two state paths could reuse offsets even when the visual relationship between cards and edges had changed.

## Implementation Notes

- `src/revisionGraph/panel/state.ts` now builds a hashed scene layout key prefixed with `fanout-balance-v1`.
- The key includes node hash, row, rounded default position, dimensions, and sorted edge topology.
- Metadata-patched graph refresh fingerprints now rebuild node layouts and compute the same topology-aware scene key as a full ready-state rebuild.
- Webview offset preservation still works when the repository path and scene layout key stay identical.
- Stale offsets are invalidated when the topology or layout-rule version changes, making normal refresh paths converge with the post-fetch disposition.

## Acceptance Criteria

- Scene layout keys are stable for the same node layout and edge topology.
- Scene layout keys change when edge topology changes, even if node rows and positions are otherwise the same.
- Metadata-patched refresh fingerprints use the same layout identity rules as full ready-state rebuilds.
- Existing graph context preservation remains intact for unchanged layouts.
