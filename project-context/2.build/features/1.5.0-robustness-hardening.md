# 1.5.0 Robustness Hardening Plan

## Approval

Status: scope frozen and Build authorized by the maintainer on 2026-06-27. Slices A through I are implemented locally; cross-platform CI execution and manual Extension Development Host validation remain Deliver evidence.

Approved architecture decision on 2026-06-27: Slice B will use one mutation lease per repository and reject overlapping mutations instead of queueing them. The maintainer subsequently approved the full bounded slice plan and authorized Build on the same date.

This artifact translates the `1.5.0` reliability proposal into bounded implementation slices. Each slice requires focused acceptance evidence and a clean handoff before the next dependent slice begins.

## Goal

Prevent stale, duplicated, unbounded, or misleading operations across Git mutations, repository lifecycle changes, filesystem restore, worker layout, Git history parsing, and ref-backed diff content.

## Architectural Constraints

- Preserve the singleton revision graph editor and on-demand Compare Results and Show Log panels.
- Prefer `vscode.git` for supported repository mutations and targeted Git CLI only where required.
- Preserve multi-repository and zero-repository behavior.
- Do not add a persistent data store, new webview, command, setting, or dependency without separate approval.
- Keep browser-side messages untrusted and revalidate authority in the extension host immediately before mutation.

## Slice Plan

### Slice A: asynchronous message boundary

Ownership:

- `src/revisionGraph/controller.ts`
- `src/compareResultsView.ts`
- `src/showLogView.ts`
- a small shared host-boundary helper if extraction reduces duplication

Work:

- Contain every `onDidReceiveMessage` promise.
- Normalize unexpected error reporting and loading-state recovery.
- Ignore expected cancellation and stale-operation errors without user noise.
- Emit diagnostics without exposing secrets or full repository content.

Tests:

- rejected handler in all three panels;
- cancellation and disposed-panel cases;
- panel remains usable after failure.

### Slice B: mutation coordinator and operation context

Ownership:

- `src/revisionGraph/messageDispatcher.ts`
- `src/revisionGraph/refActionWorkflow.ts`
- `src/revisionGraph/currentHeadWorkflow.ts`
- `src/refActions/types.ts`
- focused helper module for mutation leases/context

Work:

- Acquire one mutation lease per repository and reject overlapping mutations.
- Capture repository path plus generation; revalidate after prompts and before Git.
- Invalidate leases when a repository closes, selection changes where relevant, or the controller disposes.
- Keep read-only compare, log, clipboard, and projection operations outside the mutation lock.

Tests:

- double dispatch reaches Git once;
- different repositories can proceed independently;
- switch/close during confirmation prevents mutation;
- cancellation releases the lease;
- failed mutation releases the lease and refresh state remains coherent.

Tests:

- active and unrelated repository closure;

### Slice D: bounded Git execution policy

Ownership:

- `src/gitExec.ts`
- `src/workbenchRefActionServices.ts`
- remaining targeted Git CLI callers

Work:

- Inventory every Git CLI call as read, local mutation, or remote-capable operation.
- Apply 15 seconds/1 MiB to short metadata reads.
- Apply 60 seconds/4 MiB to local mutations.
- Apply 120 seconds/4 MiB to remote-capable CLI operations.
- Preserve reviewed specialized limits for graph history, revision logs, unified diffs, commit details, ref content, and binary restore content.
- Enforce a 60 second/4 MiB fallback when a caller omits explicit options so no Git CLI call is unbounded.
- Bound stdout/stderr, time, cancellation, and error detail.
- Settle exactly once for spawn error, close, abort, timeout, or output limit.
- Evaluate process-tree termination separately for supported platforms.

Approved decision on 2026-06-27: these profiles are the required Slice D policy. Existing `vscode.git` API operations remain on the built-in API path.

Tests:

- every settlement path;
- output caps across stdout and stderr;
- timeout/cancel race;
- operation profile coverage for every call site.

### Slice E: safe restore semantics

Ownership:

- `src/compareResultRestore.ts`
- `src/workbenchRefActionServices.ts`
- `src/compareResults/restoreAction.ts`
- `src/showLog/fileRestoreAction.ts`

Work:

- Use targeted `git restore` for tracked-file content and metadata.
- Use direct filesystem removal only when the selected source revision has no corresponding path.
- Reject or safely handle symbolic links and symlinked ancestors.
- Preserve rename, addition, deletion, executable-bit, and binary-file behavior.
- Revalidate the selected repository and source ref immediately before write/delete.

Approved decision on 2026-06-27: this hybrid restore strategy is the implementation direction for Slice E.

Tests:

- outside-target file and directory symlinks;
- broken symlinks;
- executable and binary files;
- rename combinations and partial-failure reporting.

### Slice F: NUL-safe Git history protocol

Ownership:

- `src/revisionGraph/source/graphGit.ts`
- `src/revisionGraph/backendServices/snapshot.ts`
- `src/revisionGraph/backendServices/revisionLog.ts`

Work:

- Change Git format arguments and parsing atomically.
- Use NUL framing for fields and records that cannot collide with commit content.
- Preserve short-stat parsing, decorations, ranges, pagination, and filtering.
- Fail explicitly on structurally malformed output rather than manufacturing refs or commits.

Tests:

- `U+001E`, `U+001F`, multiline body, empty fields, unusual author text, merge parents, decorations, and stats.

### Slice G: layout worker resilience

Ownership:

- `src/revisionGraph/layout/d3DagSugiyamaLayoutWorkerHost.ts`
- `src/revisionGraph/layout/layeredLayout.ts`
- graph load diagnostics and focused tests

Work:

- Settle zero exit without result and add a bounded worker timeout.
- Preserve abort semantics and terminate the worker deterministically.
- Permit synchronous d3-dag fallback only when the projection has at most 200 nodes and at most 300 edges.
- Above either threshold, never repeat d3-dag in the extension host; use a deterministic `O(V+E)` simplified fallback.

Approved decision on 2026-06-27: these node and edge thresholds define the Slice G synchronous fallback boundary.

Tests:

- result, error message, worker error, non-zero exit, zero exit without message, timeout, abort, exact threshold boundaries, and deterministic large-graph fallback behavior.

### Slice H: explicit ref content failures

Ownership:

- `src/refContentProvider.ts`
- diff-opening adapters if error presentation needs context

Work:

- Distinguish a valid empty blob from repository, ref, path, timeout, and output-limit failure.
- Surface actionable errors and avoid rendering failed content as a valid empty file.

Tests:

- empty blob success;
- stale ref/path;
- closed repository;
- timeout and output limit.

### Slice I: delivery hardening

Ownership:

- CI workflow, package metadata, release artifacts, README/CHANGELOG, and dependency lockfile only after approval

Work:

- Add a verification-only GitHub Actions matrix for Ubuntu, Windows, and macOS using Node.js 20.
- Run `npm ci`, `npm run build`, `npm test`, `git diff --check`, `npm audit --omit=dev`, and applicable platform fixtures in each matrix job.
- Use read-only repository permissions and do not package a release VSIX, change versions, create releases/tags, publish, or expose Marketplace credentials.
- Update development dependencies and `package-lock.json` to remediate audit findings without changing runtime dependencies.
- Do not use `npm audit fix --force`; stop for separate approval before any major dependency upgrade.
- Re-run clean install, build, full tests, production and development audits, and package-content inspection after dependency remediation.
- Add a versioned deterministic benchmark generator with a fixed seed and manifest.
- Generate the CI tier with 1,200 commits, 120 branches/refs, 40 merges, and 30 tags.
- Generate the release-candidate tier with 12,000 commits, 600 branches/refs, 200 merges, and 200 tags.
- Record Git load, projection, worker, fallback, host payload, and webview rendering measurements for the release-candidate manifest.
- Generate a shared Git fixture covering spaces, Unicode, option-like names, nested paths, renames, binary and empty files, executable files, and `U+001E/U+001F` commit text.
- Run Linux/macOS cases for outside and broken symlinks plus executable-bit preservation.
- Run Windows cases for external junctions, supported symlinks, drive letters, separators, long paths, spaces, and Unicode.
- Use a shared Node helper for process-tree, timeout, cancellation, and stdout/stderr flood behavior.
- Run the resulting cross-platform and Extension Development Host matrices without silently skipping mandatory release cases.
- Re-run Graphify and inspect changes to centrality and articulation points.
- Freeze scope, bump to `1.5.0`, package, install, and publish only through explicit Deliver approvals.

Approved decisions on 2026-06-27: this generated fixture set is the required cross-platform verification baseline; GitHub Actions will execute it on Ubuntu, Windows, and macOS with Node.js 20 as verification-only CI; development-only audit remediation may update the lockfile without `--force`, runtime dependency changes, or unapproved majors; and deterministic CI/release-candidate graph benchmarks use the approved fixed sizes and seed.

## Critical Path

`A → B → C → D → E → F/G/H → I`

- `F`, `G`, and `H` can proceed independently after the host action and Git execution contracts stabilize.
- `E` depends on `D` because the approved tracked-file restore path uses targeted Git CLI execution.
- No release candidate should be cut until A through H meet their acceptance criteria.

## Definition of Done per Slice

- Focused tests cover success, cancellation, failure, stale state, and disposal where applicable.
- `npm run build`, focused tests, `npm test`, and `git diff --check` pass.
- The feature artifact records changed files, behavior, known gaps, and manual verification.
- No unrelated refactor or user-visible feature is included.
- Any dependency, contribution point, or release action has explicit maintainer approval.
