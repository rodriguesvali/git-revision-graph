# 1.5.6 Architecture Risk Reduction

Status: Complete
Last updated: 2026-07-01
Target version: `1.5.6`

## Goal

Reduce maintenance risk in the revision graph and Git workflow implementation without changing the
published product surface. This patch release should make future feature work safer by narrowing
high-coupling seams identified by the updated graph analysis.

## Baseline

- Current package version: `1.5.6`.
- Primary surface remains the Source Control-launched editor revision graph.
- No new VS Code contribution points are planned.
- Existing commands, menus, view IDs, compare flows, Show Log, graph scopes, focus modes, minimap,
  search, selection, and Git workflows should remain behaviorally compatible.

## In Scope

### 1. Split Workbench Ref Action Services

Refactor `src/workbenchRefActionServices.ts` into smaller VS Code adapter modules while preserving
the existing `createWorkbenchRefActionServices()` factory.

Candidate ownership:

- Workbench UI prompts and notifications.
- Diff presenter wiring.
- Reference mutation manager.
- Compare Results fallback wiring.

Acceptance criteria:

- Existing ref action workflows continue to receive a `RefActionServices` object.
- User-facing prompts and confirmation labels remain unchanged unless a test reveals inconsistency.
- Existing ref action tests pass without weakening assertions.

### 2. Narrow Ref Action Workflow Dependencies

Where practical, update ref action workflows to depend on narrower service slices instead of the
full `RefActionServices` aggregate.

Acceptance criteria:

- Workflows that only need UI and reference mutations no longer require unrelated presenters.
- Test harnesses can supply smaller fakes for touched workflows.
- `RefActionServices` remains available as an integration aggregate for current callers.

### 3. Separate Revision Graph Message Parsing From State Authorization

Split `src/revisionGraph/messageValidation.ts` responsibilities so payload validation and
state/current-repository authorization can evolve independently.

Acceptance criteria:

- Message payload validation remains bounded and rejects malformed webview input.
- State authorization rules remain covered by existing dispatcher/validation tests.
- No webview message type is added, removed, or renamed in this release.

### 4. Document Manual Release Smoke Matrix

Create a fixed Extension Development Host smoke matrix for release-candidate validation.

Acceptance criteria:

- Matrix covers zero-repository, single-repository, multi-repository, graph refresh, repository
  switching, Compare Results, Show Log, diff open, checkout/branch/merge guards, conflict handoff,
  sync/fetch, and packaging/install checks.
- Matrix is referenced from `project-context/3.deliver/release.md`.

Implementation status:

- Added `project-context/3.deliver/extension-host-smoke-matrix.md`.

### 5. Extract Revision Graph Repository Lifecycle Coordination

Extract a focused internal coordinator from `src/revisionGraph/controller.ts` for repository
attachment, repository-set changes, repository state signatures, and refresh request creation.

Candidate ownership:

- Repository event subscription attach/detach.
- Current repository selection transitions.
- Repository set change handling.
- Repository state signature tracking.
- Current-repository refresh request construction.
- Pending follow-up refresh bookkeeping if it can be moved without changing behavior.

Acceptance criteria:

- `RevisionGraphController` remains the public owner of the graph webview surface.
- Existing graph launch, repository switching, zero-repository, and multi-repository behavior are
  unchanged.
- Refresh suppression and repository event handling keep the same semantics.
- No command IDs, view IDs, webview message types, menus, settings, or activation behavior change.
- Focused repository selection/state-change/controller tests pass.

## Out Of Scope

- New user-facing commands, menus, views, settings, or activation events.
- Additional version bumps, VSIX packaging, Marketplace publishing, or release deployment.
- Full `RevisionGraphController` redesign beyond the repository lifecycle/refresh coordination
  extraction.
- Full-history graph rendering or unbounded history loading.
- Replacing the current Git integration model or adding a custom SCM provider.
- New runtime dependencies.

## Risks

- Service extraction could accidentally change subtle VS Code prompt behavior.
- Narrower service contracts could create churn if applied too broadly.
- Message authorization split could drift from existing dispatcher expectations.
- Repository lifecycle extraction could alter multi-repository refresh timing or stale repository
  guards if the controller boundary is moved too aggressively.
- Manual smoke matrix still depends on a disposable fixture repository and maintainer time.

## Rollback

- Keep refactor commits separate from documentation/release-governance commits.
- If a regression appears in Git workflow behavior, revert the affected service extraction first and
  retain the smoke matrix documentation.
- If repository switching, empty-state handling, or refresh suppression regresses, revert the
  lifecycle coordinator extraction independently from the completed service/message split.
- Do not publish `1.5.6` until automated and manual release gates are complete.
