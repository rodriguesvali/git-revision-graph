# 1.5.5 Focus Descendants

## Status

Approved for development. Not implemented.

## Objective

Let a user select one visible revision and temporarily focus the graph on that revision plus
every loaded reference and structurally necessary path that descends from it.

## User Scenario

In a repository with many branches, the user selects the commit where a feature, release, or
integration line began and chooses `Focus Descendants`. History older than the anchor and
unrelated branches disappear, while descendant branches, tags, remotes, stashes, forks, merges,
and connecting paths remain navigable.

## Functional Scope

- Add `Focus Descendants` to the single-revision graph context menu for referenced and visible
  unreferenced commits.
- Resolve the selected revision to an anchor hash in the loaded graph snapshot.
- Keep the anchor and every loaded commit that has the anchor in its ancestry.
- Run the descendant candidate set through the existing major-operations projection so
  intermediate commits continue to collapse into `through` edge paths.
- Preserve current tag, remote, stash, and merge visibility options inside the focused result.
- Reproject and relayout from the reusable snapshot without a new Git history query when the
  snapshot compatibility boundary allows it.
- Show the active anchor in a removable, accessible toolbar indicator.
- Clear descendant focus when the user exits it or changes the main graph scope.
- Keep `Focus Descendants` and `Focus Range` mutually exclusive; activating one clears the other.
- Avoid a redundant focus action when the selected revision is already the active anchor.

## Semantic Rules

- A commit is a descendant when following its Git parent links can reach the anchor.
- Visual row, lane, timestamp, and screen position do not define membership.
- The anchor is always included and acts as the focused graph's lower boundary.
- References whose target commit is not a descendant of the anchor are excluded.
- Results are bounded by the currently loaded snapshot; this mode does not claim unbounded
  repository-history coverage.
- If a stale or invalid anchor cannot be resolved, preserve the last valid graph state and
  provide short actionable feedback rather than silently broadening the projection.

## Non-Goals

- Loading unbounded history or issuing a dedicated descendant Git query.
- Filtering by what is visually above the anchor.
- Combining Focus Range and Focus Descendants in the same projection.
- Replacing d3-dag layout, indexed virtualization, the minimap, or the graph renderer.
- Adding commands, contribution points, settings, dependencies, or product surfaces.
- Packaging or publishing `1.5.5`.

## Acceptance Criteria

- Focusing a visible anchor removes older and unrelated history.
- The anchor remains visible and every displayed descendant is reachable from it by Git ancestry.
- Descendant references and necessary structural nodes retain correct connecting edges.
- Intermediate commits continue to use the existing collapsed-path behavior.
- An anchor near a common root may legitimately produce little or no graph reduction.
- Clearing focus restores the ordinary projection without reloading Git history when the snapshot
  remains reusable.
- Main scope changes clear active descendant focus.
- Focus Range and Focus Descendants replace each other predictably.
- Search, selection, context menus, minimap, zoom, persisted offsets, and virtualized rendering
  remain correct for the focused scene.
- Zero-repository, repository switching, and retained-panel behavior remain safe.

## Test Strategy

- Unit-test descendant collection for linear, forked, merged, disconnected, and boundary graphs.
- Test projection output for anchor inclusion, unrelated-reference exclusion, structural path
  retention, visibility options, and collapsed edges.
- Test option normalization, validation, projection-only snapshot reuse, scope clearing, and
  mutual exclusion with revision ranges.
- Test webview message construction, context-action state, accessible indicator text, clearing,
  and retained-state restoration.
- Run `npm run build`, `npm test`, and `git diff --check`.

## Manual Validation

- Open a branch-heavy repository and focus from a recent fork point.
- Confirm descendant local, remote, tag, and stash references follow current visibility toggles.
- Confirm a branch that diverged before the anchor and never contains it is excluded.
- Confirm merged and unmerged descendant branches retain understandable paths.
- Switch between Focus Range and Focus Descendants and confirm only one mode remains active.
- Change the main graph scope, clear focus, switch repositories, reopen the retained panel, and
  exercise search, minimap, zoom, selection, and context menus.

## Risks

- Git ancestry through merges can produce a wider result than users expecting only first-parent
  descendants; the UI copy must describe ancestry rather than visual position.
- Old anchors may include most loaded references and provide little reduction.
- Snapshot bounds can omit descendants outside the loaded history window.
- Projection-option drift could accidentally force a full Git reload or allow simultaneous focus
  modes unless validation and compatibility tests are explicit.
- Viewport restoration may position the anchor poorly after the focused scene changes size.

## Open UX Decisions

- Place the anchor near the bottom of the viewport or center it after activation.
- Use `Descendants` or the compact existing `Focus` caption in the toolbar indicator.
