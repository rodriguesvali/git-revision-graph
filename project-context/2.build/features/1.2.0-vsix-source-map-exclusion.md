# 1.2.0 VSIX Source Map Exclusion

## Goal

Exclude generated source maps from the published extension package while preserving them in local development builds.

## Context

The existing `.vscodeignore` used `*.map`, which did not match source maps nested under `out/` or runtime dependencies. A clean `vsce ls` baseline still included:

- 694 package file entries;
- 121 files ending in `.map`;
- 117 generated source maps under `out/`, totaling 327,540 uncompressed bytes.

Source maps are useful locally, but the extension does not require them at runtime.

## Scope

- Change the `.vscodeignore` source-map rule from `*.map` to `**/*.map`.
- Keep TypeScript `sourceMap` generation enabled.
- Validate package contents with `vsce ls` without generating or publishing a VSIX.
- Update `1.2.0` release artifacts.

## Non-Goals

- No change to `tsconfig.json` or local source-map generation.
- No bundling, minification, dependency, or runtime behavior changes.
- No VSIX generation or Marketplace publication.
- No generalized package-content regression test; that remains the next package-hygiene item.

## Acceptance Criteria

- Recursive `.map` files are absent from `vsce ls`.
- Local TypeScript builds continue producing source maps in `out/`.
- Runtime JavaScript and the layout worker remain listed for packaging.
- `npm run build`, `npm test`, and `git diff --check` pass.

## Manual Validation

- No product workflow changes require an Extension Development Host smoke test.

## Risks

- Packaged stack traces will not resolve through shipped source maps. Runtime JavaScript paths and line numbers remain available, while local development keeps full maps.
- A broad recursive rule could exclude third-party maps that might aid debugging, but none are required for extension execution.
