# 1.5.7 Small Improvements

Status: Open
Last updated: 2026-07-03
Target version: `1.5.7`

## Goal

Deliver a set of small, scoped improvements to the published extension while preserving the current
architecture, native VS Code ergonomics, and release reliability.

## Baseline

- Package version opened at `1.5.7` from the published `1.5.6` baseline.
- The first approved improvement removes redundant workspace reset options from the graph context
  menu while preserving commit reset workflows.
- Packaging and Marketplace publication are out of scope until explicitly authorized.

## Scope And Acceptance Criteria

### Remove Workspace Reset Options From The Graph Context Menu

Feedback indicates that the graph actions `Reset Workspace to HEAD` and `Reset Workspace and Remove
Untracked Files` duplicate Source Control workflows. They currently appear only on `HEAD` when the
workspace is dirty.

Acceptance criteria:

- Do not show either workspace reset option in the revision graph context menu.
- Preserve `Reset to this` for commits and its existing reset-to-commit workflow.
- Preserve the existing internal workspace-reset message and host workflow; this change is limited
  to menu presentation.
- Keep stash, abort merge, and other `HEAD` context-menu actions unchanged.
- Cover the removed labels with a revision graph webview regression assertion.

### Add Push Modes To The Toolbar Long-Press Menu

Make the graph toolbar `Push` action follow the interaction already used by `Reload Revision Graph`:
a short click performs a normal push, while a 500 ms press opens a menu directly below the button.

Acceptance criteria:

- A short click on `Push` requests the `normal` push mode without opening the native mode picker.
- A long press opens a menu below the button with `Push with Force With Lease` and
  `Push with Force`.
- Selecting a force mode preserves the existing host-side destructive confirmation and Git API
  execution semantics.
- Reject missing or unknown push modes at the webview message boundary.
- Keep current-branch/upstream eligibility, multi-repository authorization, refresh behavior, and
  remote error handling unchanged.
- Cover short click, long press, menu placement, message validation, and workflow mode forwarding.

### Clarify The Reset Zoom Icon

Replace the toolbar reset-zoom glyph, which could be mistaken for a clock, power, or alert symbol,
with a compact `1:1` vector mark representing actual size / 100% zoom.

Acceptance criteria:

- Keep the icon between zoom out and zoom in and preserve the existing button behavior, tooltip,
  keyboard shortcut, and accessible label.
- Render the mark as inline SVG using the existing toolbar stroke and theme color.
- Keep it legible at the existing 16 px icon size without adding an image asset or dependency.
- Cover the new glyph and removal of the ambiguous circular glyph with a webview shell regression
  assertion.

### Subsequent Improvements

Add each further approved improvement to this artifact before or alongside implementation, including:

- User problem and intended outcome.
- In-scope and out-of-scope behavior.
- Acceptance criteria and regression risks.
- Automated and manual verification appropriate to the affected surface.

## Release-Wide Constraints

- Preserve existing repository conventions and extension architecture.
- Preserve multi-repository and empty-state behavior.
- Preserve conflict guards and native workbench ergonomics for Git mutations.
- Keep contribution points, registrations, documentation, and tests aligned when touched.
- Avoid dependency changes, destructive actions, packaging, and publication without explicit
  maintainer approval.

## Verification Plan

- Run `npm run build` after implementation changes.
- Run `npm test` for behavior changes.
- Run `git diff --check` before release-candidate packaging.
- Exercise affected workflows in an Extension Development Host when changes are user-visible.
