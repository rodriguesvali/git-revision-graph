# 1.2.0 On-Demand Command Activation

## Goal

Stop activating Git Revision Graph after every VS Code startup and rely on the extension's contributed commands to activate it when the user invokes a product workflow.

## Context

The extension targets VS Code `^1.90.0`. VS Code has inferred activation events for commands declared in `contributes.commands` since version `1.74.0`, so `onStartupFinished` is unnecessary for the five contributed commands.

Activation still runs the existing initialization path before the selected command handler executes. This slice intentionally keeps the legacy graph visibility context reset inside `activate`; removing that context and its helper remains separate cleanup work.

The compatibility alias `gitRefs.openRevisionGraph` is registered at runtime but is not contributed in the manifest. It remains available after activation, but it is not treated as an initial activation trigger.

## Scope

- Remove `onStartupFinished` from `package.json`.
- Keep the existing contributed command set unchanged.
- Replace the startup-activation manifest assertion with coverage that:
  - no explicit activation event remains;
  - the engine baseline supports implicit command activation;
  - all five user-facing activation commands remain contributed.
- Update release notes and readiness evidence.

## Non-Goals

- No removal of `gitRefs.revisionGraphVisible`, `viewLayout.ts`, or activation-time context initialization.
- No lazy import or bundling changes.
- No command ID, menu, view, Git workflow, or webview behavior changes.
- No VSIX packaging or Marketplace publication.

- Fresh-profile validation is still required because automated tests validate manifest contracts, not the VS Code extension-host activation lifecycle itself.
