# 1.0.2 Graph Drag Release Hardening

## Goal

Prevent the revision graph pan/drag mode from staying active after the left mouse button is released.

## Context

The graph viewport pan mode started on left-button `mousedown` and ended on `window` `mouseup`. In some host/browser edge cases, the release event can be missed, leaving the viewport with active drag state and the `dragging` CSS class.

## Scope

- Centralize cleanup for viewport pan, minimap drag, and node drag state.
- End pointer-driven interactions on mouseup, window blur, dragstart, document mouseleave, and mousemove events that report the left button is no longer pressed.
- Preserve existing click suppression behavior after a real viewport drag.
- Preserve node layout persistence on completed pointer-driven node drags and avoid persistence when canceled with Escape.
- Add focused webview shell coverage for the drag cleanup hooks.

## Non-Goals

- No redesign of graph pan gestures.
- No switch to pointer events in this patch.
- No changes to layout, minimap rendering, or context menu behavior beyond cleanup safety.

## Acceptance Criteria

- Graph pan mode starts only from left-button press and clears when the active mouse interaction ends.
- If a `mouseup` is missed, blur/mouseleave/dragstart or a later mousemove with no left button pressed clears drag state.
- Focused revision graph webview coverage and full automated verification pass before release handoff.

## Manual Validation

- Open the revision graph and drag the viewport with the left mouse button.
- Release the button inside and outside the graph/webview area and confirm the cursor and pan state stop.
- Start a drag, switch focus away from VS Code, return, and confirm pan mode is not stuck.
