# 0.0.25 Large Repository Layout Hardening

## Goal
Prevent large client repositories from failing revision graph load with `Maximum call stack exceeded`.

## Scope
- Keep the existing ELK layout path for normal repositories.
- Bound persisted layout-cache entries so very large graphs do not store oversized position payloads in VS Code workspace state.
- Discard oversized persisted cache entries during restore and rewrite the persisted cache with only accepted entries.
- Render a deterministic iterative lane-layout fallback when ELK throws a maximum-call-stack error.

## Acceptance Criteria
- Large graph cache payloads are skipped from persisted workspace state while the in-memory cache can still help within the current session.
- Previously persisted oversized cache entries do not keep being restored on activation.
- ELK stack-overflow failures do not prevent graph rendering.
- `npm run build` and `npm test` pass.

## Verification
- `npm run build` passed.
- `npm test` passed with 210 tests.
- Added regression coverage for oversized persisted cache rejection.

## Risks / Follow-Up
- The fallback layout is intentionally simpler than ELK and is only used after ELK fails with `Maximum call stack exceeded`.
- Manual validation in the reported large client repository is still needed before publishing.
