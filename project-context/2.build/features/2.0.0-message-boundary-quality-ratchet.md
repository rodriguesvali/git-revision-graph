# 2.0.0 Message Boundary And Quality Ratchet

Status: Implemented; automated validation complete
Updated: 2026-07-17

## Goal

Resolve the Clean Code/SOLID review findings around the revision graph message boundary and the
quality baseline: remove monolithic high-complexity switches while ensuring reviewed exceptions
cannot retain silent slack after the code improves.

## Scope And Design

- Replace the structural-validation switch with an exhaustive validator registry keyed by every
  `RevisionGraphProtocol.MessageType`.
- Keep each message validator focused and extract projection/Flow option validation into
  `messageValidationOptions.ts`.
- Replace parallel state-authorization and repository-scope switches with one exhaustive policy
  registry. Each message policy owns its state predicate, repository scope, and loading allowance.
- Replace the handler switch with an exhaustive typed handler registry while retaining the current
  focused workflows and repository guard.
- Extend the quality gate so every explicit file/function baseline must equal the current
  measurement. A decrease now requires the baseline to be ratcheted or removed.

## Compatibility And Non-Goals

- Preserve the shared protocol, message shapes, validation bounds, state/repository authorization,
  host effects, and exhaustive TypeScript coverage.
- Add no command, menu, setting, dependency, Git behavior, version, package, or Marketplace change.
- Do not alter the default quality thresholds; only reviewed exceptions and enforcement changed.

## Acceptance Criteria

- `validateRevisionGraphMessage`, `isRevisionGraphMessageAllowedForState`, repository-scope
  authorization, and `RevisionGraphMessageHandler.handleMessage` remain below the default
  complexity threshold and need no explicit exceptions.
- Adding a protocol message without a validator, policy, or handler fails TypeScript compilation.
- All malformed-input and state/repository authorization behavior remains covered by existing
  boundary tests.
- The quality gate fails when an explicit baseline is higher than the current file-line or
  function-complexity measurement.
- Existing baseline slack is removed: three file ceilings are reduced, one obsolete function
  exception is removed, and the remaining host-message exception is reduced to its actual value.

This is an internal architecture and regression-gate change with no user-visible protocol or
Marketplace impact. If a boundary regression is discovered, the registries and option validators
can be reverted together; the stricter no-slack quality check can remain independently.
