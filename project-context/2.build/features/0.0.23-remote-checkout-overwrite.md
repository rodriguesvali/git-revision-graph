# 0.0.23 Remote Checkout Local Branch Overwrite

## Goal

Improve the remote-branch checkout workflow so a user can intentionally reuse an existing local branch name when creating a tracking branch from a remote branch, reduce direct Git CLI usage for remote-touching operations where the public `vscode.git` API already covers the workflow, correct minimap scaling for small repositories, and polish the revision graph title casing/composition.

## Users

- Developers checking out remote branches from the revision graph.
- Developers running remote branch/tag deletion and graph fetch workflows from the extension.
- Developers using the minimap on small repositories with narrow or short graph layouts.
- Developers reading the active graph title to identify the repository and branch.
- Maintainers validating checkout and branch-tracking workflows before publishing a patch release.

## Problem

When checkout is executed for a remote branch, the extension prompts for a local branch name and then creates a tracking branch. If that local branch already exists, the operation fails instead of offering a safe, explicit overwrite path.

A system review for this release also found remote-touching operations that could use the `vscode.git` API instead of direct Git CLI calls, reducing authentication and Source Control integration risk.

Manual validation of a small repository showed the minimap stretching a narrow graph to fill the overview, making the small graph appear distorted.

The native graph header also showed product and branch labels in an all-uppercase style and did not include the repository name before the branch, making multi-repository workspaces harder to scan.

## Scope

- Detect an existing local branch when creating a branch from a remote branch checkout.
- Ask for explicit confirmation before overwriting the existing local branch.
- Reset the existing local branch to the selected remote branch, check it out, and configure upstream tracking.
- Move remote branch deletion, remote tag deletion, remote-name discovery, and graph fetch without `Tags` to the `vscode.git` API.
- Keep direct Git CLI only where there is no equivalent public API in the current contract: `fetch --tags`, `ls-remote --tags`, and local branch config/reset helpers.
- Preserve minimap graph proportions by fitting the graph inside the overview instead of scaling one dimension until the minimap is filled.
- Use `Git Revision Graph` casing for the product title and compose the graph view title as `<repository>: Branch: <branch>` using real-case repository and branch names.
- Preserve the current behavior for new branch names, tags, local branches, commits, cancellation, conflicts, and dirty worktrees.

## Non-Goals

- Do not publish, package, or bump the Marketplace version without approval.
- Do not replace the existing `vscode.git`-first checkout architecture.
- Do not add a new command, view, setting, or webview surface.
- Do not introduce dependency changes.

## Acceptance Criteria

- Checkout of `origin/main` with local branch name `main` still creates and tracks `main` when it does not already exist.
- Checkout of a remote branch with an existing non-current local branch name prompts before overwriting.
- If the user confirms, the existing local branch points at the selected remote branch, is checked out, and tracks that remote branch.
- If the user cancels, no checkout, branch reset, or upstream update occurs.
- Existing workspace conflict and dirty-worktree guards remain in place.
- Remote branch/tag deletion use Git API push refspecs instead of direct `git push --delete` calls.
- Remote names are read from Git API repository state with a remote-ref fallback.
- Graph fetch uses `repository.fetch(...)` unless the user selects `Tags`, where CLI remains necessary for `--tags`.
- The minimap uses one uniform contain scale so small/narrow graphs are not visually stretched.
- The native graph header combines the `Git Revision Graph` container title with a real-case repository name before `Branch: <real-case branch>`.
- Automated tests cover create, confirm-overwrite, cancel-overwrite, current-branch protection, delete refspecs, remote-name discovery, fetch option routing, proportional minimap scaling paths, and title composition.

## Changes

- Added a confirmed overwrite path when remote branch checkout is creating a local tracking branch whose name already exists.
- Added a `ReferenceManager.resetBranch` adapter backed by `git branch --force <branch> <ref>` for the overwrite step that the public VS Code Git API does not expose directly.
- Preserved `repository.checkout(...)` and `repository.setBranchUpstream(...)` for checkout and upstream tracking after an overwrite.
- Blocked overwrite of the current branch from this flow with a native warning message.
- Moved remote branch deletion and remote tag deletion to `repository.push(...)` with delete refspecs.
- Moved remote-name discovery to `repository.state.remotes`, with fallback to remote refs.
- Moved graph fetch without tag fetching to `repository.fetch(...)`; `fetch --tags` remains CLI-backed because the public API does not expose a tags option.
- Changed minimap scaling from fill-style `Math.max(...)` to contain-style `Math.min(...)`, keeping one uniform scale without stretching compact graphs.
- Changed product title casing from `GIT Revision Graph` to `Git Revision Graph` across the manifest, commands, webview shell, and docs.
- Changed the dynamic graph title to include the repository name before `Branch: <branch>` without altering repository or branch case.
- Updated README and CHANGELOG entries for the user-visible behavior.

## Manual Validation

- In an Extension Development Host, open a Git repository with `origin/main` and an existing non-current local `main`.
- From `Git Revision Graph`, run checkout on `origin/main`.
- Enter `main`, confirm overwrite, and verify `main` is checked out and tracks `origin/main`.
- Repeat and cancel the overwrite confirmation to verify no mutation occurs.
- Run graph fetch with default `Prune` only and verify it completes through the Git integration.
- Run graph fetch with `Tags` selected and verify the documented CLI fallback still works.
- Delete a remote branch/tag from a test repository and verify VS Code Git authentication behavior is preserved.
- Open a small repository with a narrow graph and verify the minimap preserves node/edge proportions without exaggerated horizontal or vertical stretching.
- Open a repository whose folder and branch include lowercase or mixed-case characters and verify the native graph header reads `Git Revision Graph: <repository>: Branch: <branch>` without forcing repository or branch names to uppercase.

## Risks

- Overwriting a local branch is destructive if the branch contains unmerged local-only commits, so the confirmation text must be explicit.
- A branch checked out in the current workspace or another worktree may be protected by Git; the extension should surface that failure cleanly.
- `fetch --tags` and `ls-remote --tags` still require direct Git CLI until an equivalent public `vscode.git` API path exists.
- Very small graphs may leave empty space in the minimap after proportional fitting; this is preferred over distorted geometry.
