# 0.0.39 Show Log Compare UI Boundary Hardening

## Goal

Continue the architectural bad-practices hardening work by moving Show Log commit-compare UI boundary setup out of the editor panel provider.

## Problem

`ShowLogViewProvider` still assembled inline VS Code message adapters for Show Log commit comparisons. That kept panel lifecycle code coupled to compare-workflow feedback policy and made the provider carry host-service wiring that belongs with the action module.

## Scope

- Let `compareLoadedShowLogCommits` and `compareLoadedShowLogCommitWithWorktree` use a lazily loaded default VS Code UI adapter when no custom UI is supplied.
- Routed `ShowLogViewProvider` through the existing compare helpers without inline `vscode.window` adapter objects.
- Preserved existing behavior for:
  - ignoring unloaded or identical commit comparisons;
  - showing no-difference information messages;
  - surfacing compare failures as operation errors;
  - opening Compare Results with `source: 'showLog'` for non-empty diffs.
- Added focused tests that prove non-empty diff paths do not require a custom UI adapter.

## Acceptance Criteria

- `ShowLogViewProvider` no longer owns commit-compare UI adapter construction.
- Compare helpers remain directly testable with custom UI services.
- Successful compare paths can run without eagerly loading VS Code UI services from tests or callers.
- Focused Show Log validation and full automated verification pass.

## Verification

- `npm run clean:test && npm run build && tsc -p ./tsconfig.test.json && node --test out-test/test/showLogCommitCompare.test.js out-test/test/showLogMessageHandler.test.js out-test/test/showLogWebview.test.js` passed with 11 tests.
- Full automated verification is pending.

## Risks

- Default UI services lazily load VS Code APIs only when Show Log compare workflows need to show information or error messages.
- Manual Extension Development Host smoke testing should still confirm Show Log compare no-difference messages and compare-error feedback.
