# 0.0.24 Graph Load I/O Parallelism

## Goal
Reduce initial revision graph load latency without changing the rendered graph or Git workflow semantics.

## Scope
- Start repository ref loading and the graph `git log` command concurrently while building a graph snapshot.
- Keep the existing graph limit policy, projection options, cancellation, and snapshot parsing behavior unchanged.

## Acceptance Criteria
- Graph snapshot loading no longer waits for `repository.getRefs()` before starting `git log`.
- Ref kind mapping still uses the complete repository refs before parsing decoration labels.
- Existing abort behavior for `git log` remains in place.
- The rendered graph state remains behaviorally unchanged.

## Implementation Notes
- `loadSnapshot` now creates `refsPromise` and `stdoutPromise` before awaiting either result.
- The optimization reduces wall-clock latency when VS Code Git API ref resolution and the Git CLI log command both incur I/O waits.

## Verification
- `npm run build`
- `npm test`

## Follow-Up Candidates
- Profile ELK layout time separately from Git snapshot time.
- Consider a fast first paint for large graphs followed by a refined layout pass if ELK dominates load time.
- Consider repository-scoped timing logs behind a development/debug flag.
