# Review Hardening

## Scope
- Address the first hardening slice from the architecture/code/efficiency review.
- Keep the published Source Control/editor graph architecture unchanged.
- Avoid dependency, manifest, version, packaging, or Marketplace changes.

## Changes
- Guard repository-scoped graph webview messages against stale repository state and active graph loading.
- Change prepared repository follow-up refresh suppression so each prepared event kind is consumed once instead of suppressing repeated events for the full window.
- Add `AbortSignal` support to Show Log revision-log loading and cancel obsolete `git log` work when a new source, scope, filter, load-more request, hide, or dispose supersedes it.
- Configure targeted Git CLI execution from VS Code's `git.path` setting, with fallback to `git`, so graph/history commands follow the user's configured Git executable.
- Refresh the repository-specific project map so it reflects the current Source Control/editor graph product surface instead of the removed primary Activity Bar graph.

## Acceptance Criteria
- Stale graph messages from a previous repository or loading state do not run repository-scoped compare, checkout, merge, delete, push, reset, log, or copy actions.
- A prepared follow-up refresh suppresses the intended repository event once, then allows later events to trigger normal refresh handling.
- Obsolete Show Log `git log` requests are aborted instead of only being ignored after completion.
- Targeted Git CLI calls use the configured executable path when `git.path` is set, including the first non-empty list entry, and continue to fall back to `git` when it is unset or blank.
- Existing graph, reference action, Show Log, Compare Results, and manifest tests remain green.
