# Show Log View State Cohesion

## Goal

Continue the `0.0.39` architectural cohesion work by separating Show Log webview state mapping from shared state/cache helpers and the editor panel provider lifecycle.

## Scope

- Add `src/showLog/viewState.ts` as the owner of Show Log state-to-webview mapping.
- Move Show Log webview payload types, summary labels, empty messages, commit labels, reference labels, short-stat formatting, lane topology projection, and inline change projection out of `src/showLogShared.ts`.
- Keep `src/showLogShared.ts` focused on the internal Show Log state shape, hidden-state creation, and bounded cached-change updates.
- Route `ShowLogViewProvider` state posting through the new view-state module.
- Preserve existing Show Log summaries, counts, empty messages, reference labels, topology payloads, inline change IDs, paths, and statuses.

## Acceptance Criteria

- Hidden Show Log state maps to the current hidden webview payload.
- Target and range sources keep the current summary labels and commit-count text.
- Expanded commits keep inline file changes, statuses, merge flags, references, and lane topology.
- Filtered empty states keep the current empty message.
- Cached-change bounding remains owned by the shared state helper.
- Focused Show Log shared-state, view-state, webview, lane, and message handler tests pass.
- `npm run build` passes.
- `npm test` passes.
- `git diff --check` passes.

## Verification

- Focused Show Log shared-state, view-state, webview, lane, and message handler tests passed with 12 tests after extracting webview state mapping. This includes `npm run build` through the focused validation command.
- `npm test` passed with 327 tests after extracting Show Log webview state mapping. This includes `npm run build` through the test script.
- `git diff --check` passed after extracting Show Log webview state mapping and updating verification artifacts.

## Risks

- Webview payload compatibility can drift if state mapping changes summary labels, reference labels, inline change IDs, or topology fields.
- The provider still owns log loading, expansion, cancellation, clipboard, diff, GitHub, and reset actions; future slices may split loading/cancellation if further architectural cohesion is needed.
- Runtime integration still needs manual Extension Development Host smoke testing because message delivery depends on a live VS Code webview.
