# 2.0.0 Task Promotion PR Context

Status: Implemented; manual Extension Development Host validation pending
Updated: 2026-07-17

## Goal

Let a task created through `Start New Task` return by Pull Request to the exact feature from which
it was created, without asking the user to reconstruct or guess that relationship later.

## User Workflow

1. The user starts a task from a classified feature branch.
2. Task creation persists the feature as `branch.<task>.git-revision-graph-flow-target` before
   optional task publication.
3. After committing task work, the user selects
   `Flow Governance > Promotion PR Context` on the task branch.
4. The host fetches and verifies the mapped feature, requires its local ref to match the remote,
   and verifies that the task contains commits ahead.
5. The existing source-publication preflight publishes or pushes the task only after confirmation,
   revalidates it, and then opens the Pull Request context.

## Scope

- Reuse the existing local branch-target metadata key and graph-load path.
- Add a task-specific `promotionTargetRefName` to serializable Flow Governance branch metadata.
- Generate exactly one `task -> feature` Pull Request target when the persisted mapping resolves to
  a visible classified feature.
- Authorize task Pull Request messages only for that host-generated target.
- Reuse the blocking `Fetching remotes...` lifecycle for target and source checks.
- Reuse existing publish/push confirmations and divergence protection for the task source.

## Non-goals

- Inferring a feature from a task name or arbitrary ancestry.
- Automatically assigning a target to tasks created before target persistence.
- Automatically pulling, merging, rebasing, force-pushing, or creating a provider Pull Request.
- Adding settings, commands, dependencies, provider APIs, or version changes.

## Acceptance Criteria

- Task creation records its source feature before optional publication.
- Failure to persist the target stops task setup before publication.
- A mapped task shows `Promotion PR Context` and no feature selector.
- A mapped task cannot request context for a different feature.
- An unmapped task or a task whose mapped feature is unavailable exposes no handoff.
- The mapped feature is fetched and must be synchronized with its remote before the ahead check.
- A committed but unpushed task offers confirmed push and must pass post-push revalidation.
- Remote-ahead, divergent, unknown, missing-target, declined, or failed operations remain blocked
  through modal feedback.

## Manual Validation

- Create a task from a feature, decline initial publication, commit once, invoke Promotion PR
  Context, confirm publication, and verify the generated target is the original feature.
- Repeat with an already published task containing one unpushed commit and confirm the normal push.
- Create two active features and verify the task still targets only its recorded source feature.
- Make the local feature behind, ahead, and divergent from its remote; each case must block before
  task publication or context generation.
- Verify a task created before this change has no inferred target and no Flow Governance submenu.

## Release-Note Impact

Document that tasks created by Flow Governance retain their source feature and provide a guarded
Promotion PR Context handoff after task work is committed.
