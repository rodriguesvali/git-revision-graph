# 0.0.24 Checkout Metadata Patch Efficiency

## Goal
Make local branch checkout and local branch deletion updates faster by avoiding full graph snapshot reloads when only ref metadata changes.

## User Workflow
Users check out another local branch or delete a local branch from the graph and expect the already-rendered graph to update ref decorations quickly.

## Scope
- Keep checkout local refreshes on the existing repository event path.
- Preserve metadata patch behavior when the current HEAD can be resolved in the loaded snapshot.
- Apply same-layout metadata patches in the webview by replacing only node ref markup, leaving graph edges, viewport, and node placement intact.
- Use an optimistic reference-removal overlay after confirmed local branch deletion so the deleted branch line disappears immediately without reloading refs or merge-blocked metadata.
- Keep metadata patch behavior as the repository-event fallback for ref metadata changes that are not covered by a direct action patch.
- Ignore non-HEAD refs whose tips are outside the current snapshot instead of forcing a full graph reload.
- Still fall back to a full reload when the current HEAD cannot be resolved in the visible snapshot.

## Non-Goals
- Changing Git checkout behavior.
- Changing graph snapshot loading limits.
- Adding persistent layout caching.
- Hiding real topology-changing updates such as fetch or merge.

## Acceptance Criteria
- Metadata patch skips off-snapshot non-HEAD refs.
- Metadata patch still declines when the active HEAD cannot be resolved.
- Checkout between visible local branches can update graph metadata without re-running full snapshot loading.
- Checkout between visible local branches updates the previous and new HEAD ref colors without rerendering graph edges.
- Deleting a visible local branch removes that branch line from the card through a direct reference patch and suppresses the follow-up repository state refresh.

## Test Plan

## Manual Validation
- Open a large repository graph.
- Check out another local branch whose commit is already visible in the graph.
- Verify the graph updates HEAD/ref decoration without a long `Updating revision graph...` wait.
- Verify the previous branch changes back to branch green and the checked-out branch changes to HEAD red without viewport or edge movement.
- Delete a local branch that shares a card with its remote tracking ref and verify only the local branch line disappears.
- Confirm the deletion does not show a long `Updating revision graph...` wait or rebuild the graph edges.
- Check out a branch outside the loaded snapshot and verify the graph can still fall back to a full reload.

## Release-Note Impact
- Improved checkout and local branch deletion refresh efficiency by avoiding full graph reloads for ref-only updates.

## Handoff Notes
- This addresses likely sources of multi-second `Updating revision graph...` waits after local checkout and local branch deletion. If delays remain, add timing instrumentation around metadata patch, merge-blocked target calculation, and full snapshot load.
