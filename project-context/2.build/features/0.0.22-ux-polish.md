# 0.0.22 UX Polish

## Goal

Refine the existing `GIT Revision Graph` experience so common graph workflows are easier to discover, the primary graph surface feels less crowded, and powerful ref actions feel safer.

## Users

- Developers using the graph as a daily Git navigation and compare tool.
- Maintainers validating branch, tag, publish, sync, merge, delete, diff, and log workflows before release.
- New users who have not learned the graph's right-click and Ctrl/Cmd-click interaction model yet.

## Problem

The extension has accumulated a strong set of graph-side workflows, but the UX now has signs of feature density:

- Important selection actions are mostly hidden behind context menus.
- The graph context menu is a long flat list.
- Toolbar controls compete for space with the graph canvas.
- Some successful actions, such as copying a commit hash, do not provide visible feedback.
- Empty states tell the user what is missing but do not always provide the obvious next action.

## Proposed Scope

1. Add a compact selection action bar for one-target and two-target graph selections.
2. Group and polish the graph context menu, including viewport clamping and destructive-action separation.
3. Reduce toolbar density by consolidating lower-frequency visibility toggles.
4. Improve small feedback and empty-state affordances.

## Changes

- Added a compact graph selection action bar for one selected target and two selected targets.
- Grouped the primary graph context menu with section labels, separators, viewport clamping, first-item focus, and destructive-action styling.
- Moved tag, remote branch, stash, and branching/merge visibility toggles into a compact `View` options menu.
- Added native feedback after copying a commit hash.
- Added an actionable empty-state button for choosing a repository when repositories are available.
- Fixed the graph-side choose-repository cancellation path so the webview leaves loading state after a canceled picker.
- Updated README and webview tests for the user-visible UX changes.

## Verification

- `npm run build` passed.
- `npm test` passed with 185 tests.

## Non-Goals

- Do not bump `package.json` to `0.0.22` until explicitly approved.
- Do not package or publish.
- Do not add new Git mutation workflows.
- Do not replace the current graph data model, projection, layout, controller, or webview architecture.
- Do not introduce a new dependency without explicit approval.

## Acceptance Criteria

- One selected graph target exposes visible actions without requiring a context menu.
- Two selected graph targets expose visible compare, show log, unified diff, and clear-selection actions.
- The graph context menu is grouped into inspect, compare, create/publish, branch operations, destructive, and selection areas where applicable.
- Destructive operations remain protected by existing native confirmations.
- The toolbar dedicates less visual space to low-frequency toggles while keeping search and repository actions easy to find.
- Copy commit hash has visible success feedback.
- Empty graph states include an action when the next step is clear and safe.
- Multi-repository, zero-repository, conflict guard, cancellation, and refresh behavior are preserved.

## Candidate Implementation Notes

- Keep the work inside the existing revision graph webview and controller message contract.
- Prefer browser-side UI changes in `src/revisionGraph/webview/script/*` and `src/revisionGraph/webview/styles.ts` before adding extension-host complexity.
- Add or update TypeScript message types only if a new host action is required.
- Reuse native VS Code messages for clipboard feedback and mutation outcomes.
- Align the graph context menu with the more mature menu patterns already used in `src/compareResultsWebview.ts` and `src/showLogWebview.ts`.

## Test Strategy

- Update webview shell/string tests for new action-bar, toolbar, and context-menu markup.
- Add focused tests for any new host message handling.
- Preserve existing ref action tests for checkout, branch, tag, publish, sync, merge, and delete workflows.
- Run `npm run build`.
- Run `npm test`.

## Manual Validation

- Validate graph toolbar at narrow and wide sidebars.
- Verify one-target and two-target selected states from referenced and unreferenced commits.
- Verify compare, compare with worktree, show log, unified diff, copy hash, create branch, create tag, publish, sync, merge, and delete actions still route correctly.
- Verify empty state with no repositories and with multiple repositories.
- Verify canceled QuickPick/InputBox/confirmation flows leave the graph usable.

## Risks

- Moving toolbar controls can reduce discoverability if the replacement view-options control is too hidden.
- Adding an action bar can crowd the graph if it is not compact and conditional.
- Grouped context menus must preserve existing action availability rules exactly.
- Keyboard and focus behavior in the webview needs manual validation in the Extension Development Host.
