# 0.0.24 ELK Layout Cache

## Goal
Reduce repeated revision graph load time without changing the visual layout produced by ELK.

## Scope
- Cache ELK layout positions by projected graph topology, node dimensions, edge order, and layout options.
- Reuse cached positions only when the projected graph submitted to ELK is identical for layout purposes.
- Keep the existing ELK layered algorithm and options unchanged.
- Keep the active cache bounded in memory and persist resolved cache entries in workspace state.

## Acceptance Criteria
- The first load of a topology still uses the current ELK layout.
- Repeated loads of the same projected topology reuse the exact previous positions.
- Reloading the extension host can restore resolved cache entries from workspace state.
- Cache misses remove failed layout promises so a transient ELK failure does not poison later loads.
- Automated tests cover cache hit/miss behavior.

## Implementation Notes
- `src/revisionGraph/layout/layeredLayout.ts` owns a 12-entry LRU-style cache and serialization helpers.
- Cache keys include the fixed ELK layout options, node hash, estimated node width/height, and ordered edges.
- Cached maps are cloned before returning so callers cannot mutate cached state.
- `src/extension.ts` restores the cache from `workspaceState` on activation and saves it after successful layout cache updates.

## Verification
- `npm run build`
- `npm test`

## Manual Validation
- Enable `gitRevisionGraph.traceLoading`.
- Load a large graph once and record `scene.layout.elk`.
- Trigger a graph reload for the same topology and confirm `scene.layout.elk` reports `cache=hit` and drops sharply.
- Reload the Extension Development Host and confirm the same topology can still produce a cache hit.
