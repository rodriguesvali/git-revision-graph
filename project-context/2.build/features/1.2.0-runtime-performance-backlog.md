# 1.2.0 Runtime Performance Backlog

## Status

Approved for the `1.2.0` performance cycle as the next implementation front after activation and package-output hygiene.

This artifact records planned runtime improvements only. Each item still requires a focused implementation slice, before/after measurement, automated validation, and manual Extension Development Host smoke coverage where user-visible graph or Show Log behavior changes.

## Objective

Reduce repeated extension-host, Git, and webview work during normal revision graph use while preserving the existing Source Control-launched graph surface, Git workflow semantics, multi-repository behavior, cancellation, stale-render guards, and retained webview state.

## Approved Runtime Improvement Set

1. Projection-only refresh with snapshot reuse. Implemented in `project-context/2.build/features/1.2.0-projection-only-refresh.md`.
   - Reuse the loaded graph snapshot when projection options change but the underlying Git history window does not need to be queried again.
   - Recompute projection, layout, and view state from the reusable snapshot instead of forcing a full Git-history reload.
   - Keep full rebuilds for repository switches, fetches, checkouts, merges, branch/tag/delete operations, or any state change that can move visible topology outside the current snapshot.

2. Request-scoped ref reuse. Implemented in `project-context/2.build/features/1.2.0-request-scoped-ref-reuse.md`.
   - Resolve repository refs once within a graph-load request and pass that stable ref set to snapshot parsing, overlays, view-state construction, and follow-up analysis that belongs to the same request.
   - Avoid introducing a long-lived ref cache; freshness should remain bounded to the active request so repository state changes remain observable.

3. Reduced Git ancestry process fan-out. Implemented in `project-context/2.build/features/1.2.0-merge-ancestry-fanout-reduction.md`.
   - Continue using the loaded commit graph for merge-blocked target analysis whenever the relevant refs are present in the snapshot.
   - Minimize fallback `git merge-base --is-ancestor` calls for unresolved refs by caching request-local ancestry answers and aggregating or short-circuiting checks where practical.
   - Include HEAD/current target identity in any ancestry cache boundary so checkout, fetch, merge, or ref movement cannot reuse stale answers.

4. Indexed virtual viewport candidate selection. Implemented in `project-context/2.build/features/1.2.0-indexed-graph-virtualization.md`.
   - Keep the current webview virtualization behavior, but add a row/range/spatial index so scroll frames inspect nearby graph elements instead of filtering the full node and edge lists on each frame.
   - Preserve minimap, search, selection, context menus, keyboard navigation, zoom, persisted offsets, and retained-panel restoration.

5. Incremental Show Log updates.
   - Keep the full-state path for initial render and recovery, but introduce smaller update messages where they materially reduce work.
   - Candidate incremental paths include append-only load-more results, per-commit expansion updates, loading indicator updates, and cached changed-file reuse.
   - Preserve filtering, source-token cancellation, commit comparison selection, context-menu availability, inline file actions, and Compare Results routing.

## Sequencing

1. Implement projection-only refresh with snapshot reuse. Complete.
2. Fold request-scoped ref reuse into the graph-load path. Complete.
3. Reduce merge-blocked ancestry fallback process fan-out. Complete.
4. Add indexed webview virtualization candidate selection. Complete.
5. Add incremental Show Log update messages where measurement justifies the complexity.

The first three items should be treated as one backend-oriented performance front with separate commits and artifacts. The webview indexing item is complete; the Show Log item should follow after viewport behavior is validated.

## Measurement Plan

- Capture `gitRevisionGraph.traceLoading` output for a fixed repository and option set before and after each graph-loading slice.
- Record full rebuild versus projection-only refresh counts for scope/filter option changes.
- Record repository ref-read counts per graph-load request.
- Record Git process counts for merge-blocked ancestry analysis.
- Record webview virtual render timings and visible node/edge counts during scroll and zoom on a large scene.
- Record Show Log host-to-webview payload sizes for initial load, load-more, filter reset, and commit expansion.

## Acceptance Criteria

- Projection-only changes do not call Git history loading when the current snapshot is valid for the requested projection.
- Request-scoped ref reuse does not hide fresh refs after fetch, checkout, branch creation, tag creation, merge, or deletion workflows.
- Merge-blocked action states remain correct for visible refs and unresolved off-snapshot refs.
- Graph scrolling and zooming remain visually correct while reducing full-scene scans in viewport frames.
- Show Log still supports cancellation, filtering, paging, commit expansion, file actions, commit comparison, and retained editor panel behavior.
- `npm run build`, `npm test`, and `git diff --check` pass for each implemented slice.

## Risks

- Snapshot reuse can show stale topology if invalidation boundaries are too broad.
- Request-local ref reuse can mask mid-request repository changes if stale results are applied after a newer render request wins.
- Ancestry caching can incorrectly enable or disable merge actions if keyed without HEAD/current target identity.
- Indexed virtualization can regress edge visibility, search highlights, or selection when relevant nodes are outside the DOM.
- Incremental Show Log messages can increase protocol complexity and make recovery paths harder unless full-state rendering remains available.

## Rollback

Each runtime performance item must remain independently reversible. If a slice regresses correctness or workbench behavior, revert that slice while preserving the already-approved package/activation hygiene improvements in the `1.2.0` cycle.
