# Show Log Reset To Commit

## Build Status

Build implementation complete for the `0.0.33` Show Log reset-to-commit workflow.

Version bump, package generation, and Marketplace publishing remain unapproved.

## Goal

Let users reset the current local branch to a commit shown in Show Log, then intentionally propagate that rewritten local branch to its upstream remote.

## Implementation Summary

- Added `Reset to this` to the Show Log commit context menu.
- Added `Compare with Worktree` to Show Log commit context menus when there is no active two-commit selection; with a two-commit selection, the menu shows only `Compare`.
- Added bounded `resetToCommit` message validation for Show Log webview messages.
- Reused the shared ref-action service layer so the Show Log action follows existing Git mutation prompts and refresh behavior.
- Required a current local branch before reset.
- Required a clean workspace before running `git reset --hard <commit>`.
- Confirmed the destructive reset action with the target branch and commit label.
- Refreshed the graph with a full rebuild after a successful reset.
- Exposed explicit graph branch operations for the current local `HEAD`: `Pull from <upstream>`, `Push to <upstream>`, and existing `Sync with <upstream>`.
- Let `Push to <upstream>` choose normal push, force-with-lease, or force before executing.
- Used targeted Git push modes for remote rewrites while preserving normal push through the Git API.
- Preserved Source Control handoff for conflicts or non-interactive authentication failures.

## Acceptance Criteria

- Show Log commit context menus expose `Reset to this`.
- Show Log commit context menus expose `Compare with Worktree` for unambiguous single-commit actions and expose only `Compare` while a two-commit comparison selection is active.
- Malformed or oversized reset messages are rejected.
- Reset is refused when no local branch is checked out.
- Reset is refused when the workspace has pending changes.
- Reset is confirmed before `git reset --hard` runs.
- Successful reset triggers a full graph rebuild.
- Pull and Push are offered from the graph for the current local `HEAD` when the branch has a matching upstream.
- Push lets the user choose normal push, force-with-lease, or force.
- Force push modes require a destructive confirmation before running.
- Authentication failures direct the user to Source Control without losing the reset result.

## Manual Validation

- Open Show Log for a local branch with upstream tracking.
- Right-click an older commit and choose `Reset to this`.
- Confirm the reset prompt and verify the branch moves to the selected commit.
- Return to the graph and use `Push to <upstream>` from the current `HEAD` context menu, then choose a force mode when the remote branch should be rewritten after a reset or history rewrite.
- Repeat with a dirty workspace and verify the reset is blocked.

## Release-Note Impact

Mention as a Show Log and graph workflow improvement: commits in Show Log can now reset the current local branch and compare a commit with the worktree when the action is unambiguous, while the graph exposes explicit Pull and Push actions for the current tracked `HEAD`, including force push modes for intentional history rewrites.
