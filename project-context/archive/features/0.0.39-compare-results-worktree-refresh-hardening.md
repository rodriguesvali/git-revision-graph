# 0.0.39 Compare Results Worktree Refresh Hardening

## Goal

Continue the architectural bad-practices hardening work by moving Compare Results post-restore worktree refresh policy out of the editor panel provider.

## Problem

`CompareResultsViewProvider` still loaded worktree diffs and applied comparison-state refresh rules after restore actions. That mixed panel lifecycle responsibilities with repository diff loading and state-transition policy.

## Scope

- Added `src/compareResults/worktreeRefresh.ts` as the owner for loading worktree diff changes and applying Compare Results refresh rules.
- Routed `CompareResultsViewProvider` through `refreshCompareResultsWorktreeComparison` after a successful restore.
- Preserved existing behavior for:
  - refreshing only matching worktree comparison state;
  - clearing the panel state when restored files align the worktree with the compared ref;
  - preserving the panel with updated changes when differences remain;
  - surfacing unchanged repository diff failures to callers.
- Added focused tests for matching refreshes, aligned-worktree clearing, and stale-state ignores.

## Acceptance Criteria

- `CompareResultsViewProvider` no longer owns worktree diff loading for post-restore refresh.
- Worktree refresh behavior is testable without webview panel lifecycle.
- Existing restore, message dispatch, state mapping, and rendering behavior remain unchanged.
- Focused Compare Results validation and full automated verification pass.

## Risks

- Post-restore refresh still depends on the provider to repost/close the panel after receiving the refresh outcome.
- Manual Extension Development Host smoke testing should still confirm Compare Results restore closes the panel when the worktree becomes aligned and keeps it open when differences remain.
