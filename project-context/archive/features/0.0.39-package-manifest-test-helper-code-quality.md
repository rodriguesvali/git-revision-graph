# 0.0.39 Package Manifest Test Helper Code Quality

## Goal

Keep package manifest tests readable by loading and typing `package.json` through one local helper.

## Problem

`test/packageManifest.test.ts` repeated `JSON.parse(readFileSync(...package.json...))` and narrow inline manifest types in each test. That made the manifest test setup noisy and increased the chance of type drift as new contribution checks are added.

## Scope

- Added a local `PackageManifest` type that covers the manifest fields asserted by the test suite.
- Added `loadPackageManifest()` in `test/packageManifest.test.ts`.
- Reused the helper across package manifest tests while preserving the existing assertions.

## Acceptance Criteria

- Manifest tests read `package.json` through one helper.
- Existing manifest contribution assertions remain unchanged.
- Focused package manifest validation and full automated verification pass.

## Verification

- `npm run build` passed after sharing package manifest test loading.
- `npm run clean:test && npm run build && tsc -p ./tsconfig.test.json && node --test out-test/test/packageManifest.test.js` passed with 10 tests.
- `npm test` passed with 394 tests. This includes `npm run build` through the test script.
- `git diff --check` passed after the package manifest test helper cleanup and verification artifact updates.

## Risks

- This is a low-risk test-only cleanup. Product manifest behavior and extension contribution points are unchanged.
