# 0.0.39 Webview Local Resource Roots Hardening

## Goal

Start the architectural bad-practices hardening work by making webview local resource access explicit and least-privilege across the revision graph and review panels.

## Problem

The extension creates script-enabled webviews without setting `localResourceRoots`. VS Code defaults allow local resources from the extension install directory and current workspace folders. The current webview HTML does not load local files through `asWebviewUri`, so leaving broad implicit roots is unnecessary and makes the host/webview boundary more permissive than the runtime needs.

## Scope

- Added `src/webviewOptions.ts` to centralize script-enabled webview options.
- Set `localResourceRoots: []` for:
  - the revision graph webview surface;
  - the revision graph editor panel;
  - Compare Results;
  - Show Log.
- Preserved `enableScripts: true` and retained editor-panel context behavior.
- Added focused tests for the shared webview option helpers.

## Acceptance Criteria

- Script-enabled webviews keep existing runtime behavior.
- No webview can load local filesystem resources unless a future slice explicitly grants roots.
- Webview option policy is centralized instead of duplicated at creation sites.
- Build and focused webview tests pass.

## Verification

- `npm run clean:test && npm run build && tsc -p ./tsconfig.test.json && node --test out-test/test/webviewOptions.test.js out-test/test/revisionGraphWebview.test.js out-test/test/compareResultsWebview.test.js out-test/test/showLogWebview.test.js` passed with 28 tests.

## Risks

- If a future webview starts loading local assets, it must explicitly widen `localResourceRoots` and use `asWebviewUri`.
- Manual Extension Development Host smoke testing should still confirm all three webview surfaces render and handle actions normally.
