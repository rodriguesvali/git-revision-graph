# 1.3.0 d3-dag Edge Routes

Created: 2026-06-24

Status: Implemented.

Target release: `1.3.0`.

## Goal

Preserve d3-dag Sugiyama `link.points` through the Revision Graph layout pipeline and use those route points to draw clearer graph edges, especially around merge-heavy, fan-out, and multi-layer histories.

The slice should improve edge readability without changing the existing graph product surface, node layout model, worker-thread layout execution, virtualized rendering, minimap behavior, or native VS Code Git workflows.

## Current Behavior

- `src/revisionGraph/layout/d3DagSugiyamaLayout.ts` runs d3-dag and currently keeps only node coordinates.
- `src/revisionGraphData.ts` converts node coordinates into row/lane scene nodes and emits edges as `{ from, to }`.
- `src/revisionGraph/webview/shared.ts` and `src/revisionGraph/webview/script/bootstrap.ts` build simplified single-bend edge paths from node endpoints.
- `src/revisionGraph/webview/script/layout.ts` recomputes edge paths after horizontal node dragging by using current node centers.
- The minimap renders simple straight-line edges from current node centers.

## Scope

Included:

- Extract and serialize d3-dag route points for each projected edge.
- Preserve route metadata across the worker result and layout cache.
- Carry route metadata into `RevisionGraphScene.edges`.
- Render initial graph edges from d3-dag route points when available.
- Keep endpoint-based fallback for missing, invalid, or stale route data.
- Keep horizontal drag behavior correct by updating route points with node horizontal offsets or falling back to endpoint paths during/after drag.
- Add regression tests for route extraction, worker serialization, scene edge payloads, webview path rendering, fallback behavior, and drag recalculation.

Excluded:

- No visible layout-mode UI.
- No switch to `grid()` or `zherebko()`.
- No left-to-right orientation change.
- No minimap route rendering unless it is very low risk; straight-line minimap edges can remain as-is.
- No changes to compare, checkout, branch, sync, merge, delete, reset, or Show Log workflows.
- No dependency additions.

## Design

### Data Model

Introduce a small serializable route point type:

```ts
interface D3DagSugiyamaLayoutPoint {
  readonly x: number;
  readonly y: number;
}

interface D3DagSugiyamaEdgeRoute {
  readonly from: string;
  readonly to: string;
  readonly points: readonly D3DagSugiyamaLayoutPoint[];
}
```

Add route data to the layout result:

```ts
interface D3DagSugiyamaLayoutResult {
  readonly positions: Map<string, D3DagSugiyamaLayoutPosition>;
  readonly edgeRoutes: Map<string, D3DagSugiyamaEdgeRoute>;
  readonly profile: D3DagSugiyamaLayoutProfile;
}
```

Use a stable key such as `${from}->${to}`. If duplicate projected edges ever appear, preserve the current deduped projection behavior rather than introducing multiedge rendering in this slice.

### Coordinate Translation

d3-dag points are in the same coordinate space as node positions before `buildNodeLayouts(...)` adds final card dimensions, non-overlap shifts, vertical dynamic gaps, and `GRAPH_PADDING_TOP`.

Implemented approach:

1. Preserve raw d3-dag route points in the layout result.
2. Carry route points through worker messages, in-memory cache, persisted cache, and `RevisionGraphScene.edges`.
3. Transform route points during webview rendering by interpolating the delta between raw d3-dag endpoint centers and final card centers.
4. Keep source and target anchored at the rendered card boundaries.

### Rendering

Add optional route points to `RevisionGraphEdge`:

```ts
interface RevisionGraphEdge {
  readonly from: string;
  readonly to: string;
  readonly route?: readonly RevisionGraphEdgeRoutePoint[];
}
```

Initial render should prefer route-aware paths:

- Start at the current parent-card anchor.
- Add valid intermediate route points.
- End at the current child-card anchor.
- Use a polyline-style `M ... L ...` path first.
- Optionally use a later smoothing pass only after visual validation.

Fallback to the existing `describeEdgePath(...)` whenever route points are missing, malformed, or too short.

### Horizontal Drag

Horizontal node dragging currently stores `nodeOffsets` and recomputes every visible edge from current node centers.

For route-aware edges:

- Keep endpoint anchors live: source and target anchors must always use current node positions.
- Shift intermediate route points by an interpolated or endpoint-based horizontal delta:
  - If both endpoints moved by the same offset, shift all intermediate points by that offset.
  - If endpoints moved differently, interpolate point X shift by relative Y position between source and target.
  - Leave Y unchanged.
- If the interpolation is not stable enough, fall back to the existing endpoint path while any connected node has a nonzero offset.

Implemented first implementation:

- Route rendering is used for the default layout when valid route points are available.
- If `nodeOffsets[from]` or `nodeOffsets[to]` is nonzero, the edge uses the existing endpoint path.
- This keeps drag safe and simple. A later slice can improve offset-aware route preservation.

### Cache Identity

Implemented cache behavior:

- Bumped the layout strategy namespace to `d3-dag-sugiyama-v4`.
- Persisted route data when the cache entry remains bounded.
- Kept `PROJECTED_GRAPH_LAYOUT_CACHE_PERSIST_MAX_POSITIONS`.
- Added bounded route guards for total route count and points per route.

## Implementation Steps

1. Extract route data from d3-dag.
   - Inspect the actual `graph.links()` objects after `layout(graph)`.
   - Confirm available point property names in the installed `d3-dag` version.
   - Add route extraction with strict finite-number validation.

2. Update worker messages.
   - Serialize edge routes as arrays.
   - Update worker host to rehydrate route maps.
   - Preserve fallback-to-synchronous-layout behavior.

3. Update layout cache model.
   - Decide whether routes are persisted or recomputed on cache miss.
   - If persisted, update `SerializedProjectedGraphLayoutCacheEntry` and validation.
   - Bump the cache namespace.

4. Update scene model.
   - Add optional route points to `RevisionGraphEdge`.
   - Preserve existing `{ from, to }` semantics for all graph workflow logic.
   - Update `buildRevisionGraphScene(...)` to attach routes only for visible edges.

5. Update webview/shared render helpers.
   - Add `describeRoutedEdgePath(...)` or extend `renderEdge(...)`.
   - Keep `describeEdgePath(...)` as fallback.
   - Add tests for route-aware path output and malformed-route fallback.

6. Update webview shell runtime.
   - Update `renderEdgeMarkup(...)` to include route data in markup or use edge route points directly during render.
   - Update `buildEdgePath(...)` in `script/layout.ts` to choose route-aware paths only when no connected node offset is active.
   - Keep current endpoint fallback for drag and stale route data.

7. Update virtual edge indexing if needed.
   - `getEdgeVerticalBounds(...)` currently uses endpoint node tops.
   - If route points can extend beyond endpoint vertical bounds, include route point min/max Y in the virtual edge bounds.

8. Update tests.
   - Layout result includes route entries for a merge/fan-out projection.
   - Worker returns route entries.
   - Scene edges carry route metadata.
   - Render helper emits routed path when route metadata is valid.
   - Render helper falls back to existing path when route metadata is missing or invalid.
   - Webview shell includes route-aware runtime helpers.
   - Drag update path falls back for connected offset nodes.

9. Update artifacts.
   - Update `project-context/docs/d3-dag-performance-ux-opportunities.md`.
   - Update `project-context/docs/release-1.3.0-prioritization.md`.
   - Update `project-context/3.deliver/release.md`.
   - README only needs an update if the user-facing graph layout description changes materially.

## Acceptance Criteria

- d3-dag route points are preserved for visible projected edges where available.
- Initial edge paths use route data and remain anchored to card boundaries.
- Missing or invalid route data falls back to the current endpoint-based path.
- Horizontal drag does not create broken, stale, or detached edge paths.
- Virtualized rendering still shows edges correctly while scrolling and zooming.
- Layout cache restoration remains bounded and validated.
- `npm run build`, `npm test`, and `git diff --check` pass.

## Risks

- d3-dag route point coordinates may not map cleanly to the post-processed card layout.
- Persisting routes can increase workspace state payload size.
- Route-aware rendering can regress manual node dragging if offsets are applied inconsistently.
- More detailed paths can look noisier on small/simple graphs; keep fallback simple and validate visually.
- Cache namespace changes invalidate older persisted layout positions again.

## Open Questions
