# 1.1.0 Dirty Worktree Refresh Loop

## Goal

Stop the revision graph from entering repeated `Loading Revision Graph...` refreshes when the selected repository only has ordinary workspace changes.

## Context

The revision graph listens to `vscode.git` repository state changes. That event fires for working tree, index, untracked, and merge-conflict changes as well as for refs and checkout changes. The graph was treating every state event as a full rebuild trigger, so a repository with active workspace changes could repeatedly reload the graph and make Source Control appear to refresh every few seconds while the graph panel was open.

Workspace changes do not alter the revision graph topology. They only affect graph-side action availability such as `Reset Workspace` and `Abort Merge`.

## Scope

- Track a graph-relevant repository state signature built from `HEAD` and refs.
- Keep full graph rebuilds for state changes that alter refs or `HEAD`.
- For dirty/index/untracked/merge status-only changes, post a lightweight state update without showing graph loading.
- Preserve `Abort Merge` visibility by refreshing conflict status in the lightweight update path.
- Add focused regression coverage for the state signature and status-only update behavior.

## Non-Goals

- No change to explicit reload, fetch, checkout, branch, pull, sync, merge, delete, or reset workflows.
- No change to graph topology loading or layout logic.
- No new webview message contract.

## Acceptance Criteria

- Ordinary workspace changes do not trigger full revision graph reloads.
- Ref or `HEAD` changes still trigger full revision graph reloads.
- Dirty and merge-conflict status still update graph-side action availability.
- Automated verification passes before release handoff.
