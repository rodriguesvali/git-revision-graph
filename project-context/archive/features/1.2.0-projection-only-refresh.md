# 1.2.0 Projection-Only Refresh

## Status

Implemented for the `1.2.0` performance cycle.

## Objective

Avoid re-running Git history loading when a graph option change can be handled by re-projecting the already loaded snapshot.

## Scope

- Add a `projection-only` internal refresh intent.
- Schedule projection option changes through `projection-only` instead of unconditional `full-rebuild`.
- Reuse the last applied graph snapshot only when it belongs to the current repository and was loaded with Git-history-affecting options compatible with the target projection.
- Fall back to `full-rebuild` when there is no reusable snapshot or when options that can affect `git log` loading changed.
- Keep snapshot reuse request-local to the active controller state; clear it on repository changes and explicit empty-cache reloads.

## Safety Boundary

Snapshot reuse is allowed when these Git-history-affecting options are unchanged:

- `refScope`
- `showTags`
- `showRemoteBranches`
- `showStashes`

The projection-only path may therefore handle options such as `showMergeCommits` and normalized current-branch descendant behavior without reloading Git history. Scope/ref-decoration changes still fall back to full rebuild because they can change the loaded commit window or available decorations.

## Implementation Notes

- `src/revisionGraphRefresh.ts` now defines the `projection-only` intent, lightweight loading feedback, no follow-up repository-event suppression for projection-only requests, and the reusable-snapshot compatibility helper.
- `src/revisionGraph/viewStateWorkflow.ts` now requests `projection-only` after projection option changes.
- `src/revisionGraph/controller.ts` tracks the last snapshot only after a render result is applied, resolves the effective intent before loading feedback, and reuses that snapshot for compatible projection-only refreshes.
- `src/revisionGraph/panel/state.ts` exposes the from-snapshot state builder used by the controller reuse path.

## Risks

- A future projection option may affect Git loading but accidentally be treated as projection-only if it is not added to the compatibility helper.
- The current safe boundary intentionally leaves some possible reuse opportunities on the table for correctness.

## Rollback

Revert the `projection-only` refresh intent and controller snapshot reuse path. Projection option changes can safely return to scheduling `full-rebuild`.
