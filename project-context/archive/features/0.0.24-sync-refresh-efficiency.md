# 0.0.24 Sync Refresh Efficiency

## Goal
Reduce unnecessary full graph reloads after `Sync with...` when synchronization only changes branch metadata.

## User Workflow
Users run `Sync with <upstream>` from the revision graph and expect the graph to refresh quickly when the operation is a simple push or fast-forward pull.

## Scope
- Use `metadata-patch` after push-only sync.
- Use `metadata-patch` after pull-only sync.
- Keep `full-rebuild` for diverged sync because pull may create or reveal topology changes.
- Preserve existing sync guards for unpublished tracking, conflicts, dirty worktrees, and no-upstream states.

## Non-Goals
- Changing Git pull/push behavior.
- Inferring exact pull strategy from Git configuration.
- Removing full reload fallback when metadata patch cannot resolve the current HEAD.

## Touched Surfaces
- `src/refActions.ts`
- `test/refActions.test.ts`

## Acceptance Criteria
- Push-only sync schedules `metadata-patch`.
- Pull-only sync schedules `metadata-patch`.
- Diverged sync schedules `full-rebuild`.
- Existing sync messages and Git API calls remain unchanged.

## Test Plan
- Run `npm run build`.
- Run `npm test`.

## Manual Validation
- Open a repository graph with a published local branch.
- Sync a branch that is only ahead and verify graph metadata updates without a full reload.
- Sync a branch that is only behind and verify fast update when the new HEAD is already in the loaded snapshot.
- Sync a diverged branch and verify full graph refresh still occurs.

## Release-Note Impact
- Improved `Sync with...` refresh efficiency by using metadata patches for push-only and pull-only syncs.

## Handoff Notes
- This relies on the metadata patch fallback added for checkout efficiency. If pull-only sync advances to a commit outside the loaded snapshot, metadata patch declines and the graph can still fall back through the normal refresh path.
