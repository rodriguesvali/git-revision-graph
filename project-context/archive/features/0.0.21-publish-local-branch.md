# 0.0.21 Publish Local Branch to Remote

## Goal
Allow users to publish a local branch to a remote from `Git Revision Graph`.

## Scope
- Keep release `0.0.21` focused on this feature only.
- Support publishing a selected local branch to a selected remote.
- Configure the branch upstream during publish when the Git API supports it.
- Preserve repository selection, cancellation handling, conflict guards, confirmations, and refresh behavior.
- Use native VS Code UX for remote selection, confirmation, success, and error messages.

## Out of Scope
- Version bump, VSIX packaging, or Marketplace publishing without separate human approval.
- Publishing tags, deleting remote refs, or changing existing tag push behavior.
- Bulk branch publication.
- Replacing existing sync/push/pull behavior.
- Reworking graph rendering or repository discovery.

## Acceptance Criteria
- A user can publish a local branch to a chosen remote from the graph workflow.
- The action is only available or only proceeds for local branches where publishing is meaningful.
- The user gets a clear confirmation before pushing to a remote.
- The branch is pushed with upstream tracking configured when possible.
- Authentication or Git push failures surface actionable error messages without crashing the extension.
- Existing compare, checkout, branch creation, sync, merge, delete, tag, diff, log, and graph behavior remain unchanged.

## Verification Plan
- `npm run build`
- `npm test`
- Manual Extension Development Host smoke:
  - Open a repository with a local branch that is not on the remote.
  - Trigger the publish action from the Revision Graph branch workflow.
  - Pick a remote and confirm the publish.
  - Verify the remote branch exists and the local branch tracks it.
  - Verify cancellation from remote selection and confirmation leaves the repository unchanged.

## Changes
- Added a testable `publishLocalBranchResolvedReference` workflow in `src/refActions.ts`.
- Added the `publish-branch` graph webview message and controller handler.
- Added `Publish Branch to Remote` to the context menu for local `head` and `branch` refs.
- Reused the existing native remote picker and confirmation pattern.
- Updated README feature lists for local branch publishing.
- Added unit coverage for successful publish, remote selection, no remotes, cancellation, tracked-branch guard, conflict guard, and authentication recovery.
- Added webview shell coverage for the new context menu action.
- Fixed the publish guard so a branch with inherited upstream tracking, such as `teste01` tracking `origin/auth_version`, can still be published to `origin/teste01`.
- Added published-local-branch metadata to the graph state so `Publish Branch to Remote` is hidden after a branch tracks its matching remote branch.
- Fixed the sync action visibility and backend guard so branches with inherited upstream tracking do not offer or run `Sync with <old-upstream>`.

## Verification
- `npm run build` passed.
- `npm test` passed with 184 tests.
- Manual Extension Development Host validation passed per maintainer smoke test:
  - New local branch publish works.
  - Branches with inherited upstream tracking can be published to their matching remote branch.
  - `Publish Branch to Remote` is hidden after the branch tracks its matching remote branch.
  - `Sync with <old-upstream>` is no longer offered for branches with inherited upstream tracking.

## Decisions
- The first `0.0.21` implementation exposes the feature from the graph context menu only, not as a new Command Palette command.
- Branches already tracking the same remote branch are not republished through this action; users are directed to the existing Sync workflow.
- Branches with inherited or mismatched upstream tracking are allowed to publish, and the confirmation explains that upstream tracking will be updated to the selected remote branch.
- The webview hides the publish action for branches whose local name matches their upstream remote branch, while keeping it available for inherited or mismatched upstream tracking.
- The webview only shows `Sync` for a current branch that tracks the matching remote branch; inherited upstream tracking should route the user through publish first.
- The action uses `repository.push(remoteName, branchName, true)` from the built-in VS Code Git API so upstream tracking is configured by the Git integration.
- Dirty worktrees are allowed for branch publish because push/upstream setup does not modify the worktree; unresolved merge conflicts remain blocked.

## Implementation Notes
- Prefer the built-in VS Code Git repository API for branch push/upstream behavior.
- Add targeted helpers only where the public API does not expose required branch metadata.
- Keep `package.json`, command registration, README, tests, and graph actions aligned if a new command or webview action is introduced.
