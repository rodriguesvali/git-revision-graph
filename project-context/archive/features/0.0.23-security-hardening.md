# 0.0.23 Security Hardening

## Goal

Incorporate focused security hardening into release `0.0.23` based on the 2026-04-28 security review, without changing the published extension architecture or adding new user-facing surfaces.

## Users

- Developers using the revision graph webview to run Git workflows.
- Developers using Compare Results and Show Log webviews.
- Maintainers preparing the `0.0.23` Marketplace release.

## Problem

The security review did not find direct shell injection in Git execution paths, but it identified hardening gaps around the webview-to-extension-host boundary, branch-name validation, CSP nonce generation, and release tool dependency audit state.

## Scope

- Add runtime validation for webview messages before dispatching host-side actions.
- Validate action payloads against the current extension state where possible:
  - Graph refs and revisions should resolve to visible/current graph state before compare, checkout, branch, tag, merge, delete, push, or remote-tag-state actions.
  - Compare Results actions should continue resolving item IDs only through current in-memory result items.
  - Show Log file actions should continue resolving changes through cached commit changes, and commit-level actions should be restricted to loaded log entries.
- Strengthen branch-name validation for create/reset/tracking flows to align with Git branch ref rules.
- Switch CSP nonce generation from `Math.random()` to cryptographic randomness.
- Review Git command argument construction and add `--` separators where commands accept refs and pathspecs in the same argument domain.
- Keep direct Git CLI only for established local/history operations or API gaps already documented for this release.
- Document the `npm audit` dev-tool finding and decide whether to update packaging dependencies in a separate approved dependency-change task.

## Non-Goals

- Do not publish, package, or bump the Marketplace version without approval.
- Do not replace the existing webview architecture.
- Do not add new settings, commands, or views.
- Do not broaden the release into a dependency upgrade unless explicitly approved.
- Do not remove targeted Git CLI paths that are required for graph/history data or API gaps.

## Acceptance Criteria

- Malformed webview messages are ignored or reported without throwing uncaught errors.
- Host-side Git actions reject invalid payload shapes before reaching `refActions`, `backend`, or `workbenchRefActionServices`.
- Graph actions that mutate repository state only accept refs/revisions present in the current graph state or otherwise explicitly allowed by the action contract.
- Show Log commit details only opens for loaded commit hashes.
- Show Log file actions only open/copy/compare changes that exist in cached changes for the selected loaded commit.
- Branch name input rejects empty names, leading/trailing whitespace, `refs/heads/` prefixes, names beginning with `-`, invalid Git ref characters, path traversal-like ref segments, `.lock` suffixes, and other invalid branch ref patterns.
- CSP nonce generation uses `node:crypto`.
- Existing CSP remains restrictive: `default-src 'none'`, nonce-gated scripts, and no newly introduced external resource permissions.
- `npm run build` passes.
- `npm test` passes.
- `npm audit --audit-level=low` result is recorded in release notes, with dependency remediation either completed under approval or left as a documented dev-tool risk.

## Build Plan

1. Define validation helpers for webview messages. `Done`
   - Add small, testable runtime guards instead of relying only on TypeScript discriminated unions.
   - Keep guards near each host boundary or in focused shared modules if duplication becomes meaningful.

2. Harden Revision Graph message routing. `Done`
   - Validate `message.type` and required fields before the switch dispatch.
   - Add current-state lookup helpers for visible references and commit hashes.
   - Ensure mutating actions resolve from current graph state before calling checkout, branch, tag, publish, delete, or merge workflows.

3. Harden Show Log message routing. `Done`
   - Validate message shape before dispatch.
   - Restrict commit details to loaded entries.
   - Preserve current cached-change lookup behavior for file-level actions.

4. Harden Compare Results message routing. `Done`
   - Validate message shape and array fields before dispatch.
   - Preserve item ID resolution through current in-memory result items.

5. Add branch-name validation. `Done`
   - Introduce a branch-name validator modeled after the existing tag validator.
   - Wire it into `promptBranchName`.
   - Cover create branch and remote checkout overwrite flows.

6. Strengthen CSP nonce generation. `Done`
   - Replace `Math.random()` nonce construction with `crypto.randomBytes`.
   - Keep webview HTML structure and CSP directives unchanged unless testing reveals an issue.

7. Review Git argument separators. `Done`
   - Reviewed current targeted Git command sites.
   - No separator changes were applied because the reviewed commands either use API calls, accept fixed ref-only arguments, or use Git syntaxes like `ref:path` where inserting `--` would alter semantics.

8. Update release documentation. `Done`
   - Add hardening items to `CHANGELOG.md` and `project-context/3.deliver/release.md`.
   - Record audit status and any remaining dependency risk.

## Test Strategy

- Added unit tests for branch-name validation.
- Added tests for malformed webview messages at each host boundary.
- Added tests for rejecting graph action payloads that are not present in current graph state.
- Added tests for cryptographic nonce shape/non-reuse without asserting deterministic values.
- Added ref action coverage to ensure invalid branch names do not reach Git.
- `npm run build` passed.
- `npm test` passed with 201 tests.
- `npm audit --audit-level=low` still reports 4 moderate dev-tool dependency vulnerabilities through `@vscode/vsce -> @azure/identity -> @azure/msal-node -> uuid`; dependency remediation remains a separate approval point.

## Changes

- Added runtime validators for Revision Graph, Compare Results, and Show Log webview messages.
- Added current-state authorization for Revision Graph actions that operate on refs, revisions, commits, or remote tag state.
- Restricted Show Log commit details to loaded commit hashes and file actions to cached changes for loaded commits.
- Added `validateGitBranchName` and wired it into both native branch input validation and the branch creation action path.
- Replaced `Math.random()` CSP nonce generation with `crypto.randomBytes(16).toString('base64url')`.
- Added focused tests for branch validation, message validation, graph-state action checks, nonce generation, and invalid branch rejection before Git.

## Manual Validation

- Launch the Extension Development Host.
- Open a Git repository and verify the revision graph loads normally.
- Exercise compare, show log, checkout, create branch, create tag, merge, delete, publish, sync, and fetch workflows from the graph.
- Verify invalid branch names are rejected in the native input box before Git is called.
- Verify Compare Results and Show Log still open diffs and context-menu actions correctly.
- Verify no CSP errors appear in the webview developer tools.

## Risks

- Overly strict graph-state validation could block legitimate actions for structural commits, detached commits, or stale webview state after refresh.
- Branch validation must match Git enough to prevent invalid names without rejecting common valid names like `feature/foo`.
- Adding `--` separators requires command-by-command care because not every Git subcommand accepts separators in the same position.
- Dependency audit remediation may require package-lock churn or tooling upgrades; keep that as a separate approval point if needed.
