# 1.2.0 Request-Scoped Ref Reuse

## Status

Implemented for the `1.2.0` performance cycle.

## Objective

Avoid duplicate repository ref reads during one revision graph load while keeping ref freshness bounded to the active request.

## Scope

- Share a request-scoped refs promise between graph snapshot loading and repository overlay construction.
- Preserve the existing parallel shape where ref loading and `git log` can run concurrently.
- Keep projection-only refreshes independent: when they reuse an older snapshot, the overlay still performs a fresh ref read for that new refresh.
- Avoid adding any long-lived ref cache.

## Implementation Notes

- `RevisionGraphSnapshotLoadContext` carries optional request-scoped refs into the snapshot backend.
- `buildReadyRevisionGraphViewStateBundle(...)` now starts one strict `getRefs()` read, passes it to `loadGraphSnapshot(...)`, and reuses the same promise for the repository overlay.
- `DefaultRevisionGraphSnapshotBackend` consumes provided refs instead of calling `repository.getRefs()` again.
- Overlay tracing now records whether refs came from the request context or from a direct repository read.
- Snapshot tracing now records whether `snapshot.getRefs` used the request context or read the repository directly.

## Safety Boundary

The shared refs are scoped to one ready-state construction request only. They are not persisted in the controller, snapshot cache, or webview state as a general-purpose ref cache.

Projection-only refreshes created from a previously applied snapshot still call the overlay path with a new ref read, preserving freshness for later refreshes.

## Manual Validation Focus

- Open the revision graph in an Extension Development Host and confirm initial load still shows HEAD, local branches, remotes, tags, and stash refs correctly.
- Toggle projection options and confirm projection-only refreshes keep current refs fresh.
- Fetch or switch branches and confirm full rebuilds still reflect moved refs.

## Risks

- If future state construction paths bypass `buildReadyRevisionGraphViewStateBundle(...)`, they may not receive request-scoped refs.
- If ref reads become tolerant in the snapshot path later, the shared strict read behavior should be reviewed so full-load error semantics remain intentional.

## Rollback

Remove `RevisionGraphSnapshotLoadContext` and return snapshot loading plus repository overlay to separate `repository.getRefs()` calls.
