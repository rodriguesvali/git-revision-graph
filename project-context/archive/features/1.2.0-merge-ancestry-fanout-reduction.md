# 1.2.0 Merge Ancestry Fan-Out Reduction

## Status

Implemented for the `1.2.0` performance cycle.

## Objective

Reduce Git process fan-out while determining which visible refs should have merge disabled because they are already ancestors of the current `HEAD`.

## Scope

- Keep the existing in-memory graph analysis as the first and cheapest path.
- Replace the previous per-ref unresolved fallback with one batched Git query using `git for-each-ref --merged=<HEAD>`.
- Remove the older `git merge-base --is-ancestor` per-ref fallback from merge-blocked graph analysis.

## Implementation Notes

- `DefaultRevisionGraphMergeAnalysisBackend` still computes merge-blocked refs from the loaded snapshot whenever possible.
- For unresolved off-snapshot refs, the backend now asks Git once for merged local, remote, tag, and stash refs:
  - `refs/heads`
  - `refs/remotes`
  - `refs/tags`
  - `refs/stash`
- Visible ref names are normalized to full ref names before comparing with the merged-ref output.
- If the batched query fails, the merge-blocked target load fails like other graph-state Git failures instead of silently switching to a second implementation.

## Expected Impact

For a graph view with multiple visible refs that are outside the loaded snapshot, the common fallback path drops from one Git process per unresolved ref to one Git process for the unresolved batch.

## Verification

Focused and full-suite validation completed:

- `npm run build`
- `npm run clean:test && npm run build && tsc -p ./tsconfig.test.json && node --test out-test/test/revisionGraphBackend.test.js`
- `npm test` passed with 453 tests after removing the per-ref fallback path.
- `git diff --check` passed.

## Manual Validation Focus

- Open a repository with several local, remote, and tag refs.
- Confirm refs already merged into the current branch still omit/disable merge actions in the graph context menu.
- Confirm refs not merged into the current branch still allow merge.
- Confirm behavior remains sane in normal Git repositories where Git can answer the batched merged-ref query.

## Risks

- `for-each-ref --merged=<HEAD>` uses full refs, so name normalization must stay aligned with graph reference presentation.
- If future visible ref kinds are added, they need explicit full-ref normalization before they can participate in the batched path.
- The implementation now has a single Git fallback path; a `for-each-ref --merged` failure is no longer masked by per-ref `merge-base` checks.

## Rollback

Reintroduce the previous per-ref `merge-base --is-ancestor` checks if the batched merged-ref approach proves incompatible with supported Git environments.
