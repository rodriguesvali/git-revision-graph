# 0.0.24 Graph Load Tracing

## Goal
Provide opt-in timing diagnostics for slow revision graph loads so future performance work can target the real bottleneck.

## Scope
- Add a disabled-by-default VS Code setting: `gitRevisionGraph.traceLoading`.
- Emit graph load timing phases to the `Git Revision Graph` output channel only when tracing is enabled.
- Instrument snapshot I/O, git log loading, commit graph parsing, projection, ELK layout, primary ancestor path building, merge-blocked target resolution, and ready-state assembly.

## Acceptance Criteria
- Normal users see no load timing output by default.
- Developers can enable `gitRevisionGraph.traceLoading` and reload the graph to collect phase timings.
- The trace output includes request id, refresh intent, repository label, phase name, duration, and phase details.
- Existing graph behavior and rendering remain unchanged.

## Implementation Notes
- `src/revisionGraph/loadTrace.ts` defines the shared trace event and helper functions without depending on VS Code APIs.
- `RevisionGraphController` owns the VS Code setting lookup and output channel.
- Backend, state, and scene builders receive an optional trace sink and skip all output work when it is not provided.

## Verification
- `npm run build`
- `npm test`

## Manual Validation
- Enable `gitRevisionGraph.traceLoading`.
- Open or refresh a large revision graph.
- Inspect the `Git Revision Graph` output channel for `revision-graph-load` entries.
