# Sync Pull Refresh

## Build Status

Build implementation complete for the `0.0.30` sync pull refresh correction.

Version bump, package generation, and Marketplace publishing remain unapproved.

## Goal

Restore conservative graph refresh behavior after `Sync with...` performs a pull, so visual correctness wins over avoiding a reload when `HEAD` may advance to newly pulled commits.

## User Workflow

A user runs `Sync with <upstream>` from the revision graph. If the current branch is behind its upstream, the extension pulls and then reloads the graph from repository state.

## Scope

- Keep `metadata-patch` for push-only sync because the loaded commit topology should not change.
- Use `full-rebuild` for pull-only sync because pulled commits can change visible topology and layout.
- Keep `full-rebuild` for diverged sync, which already pulls and pushes.
- Prefer `repository.getRefs()` over potentially lagging `repository.state.HEAD.commit` when resolving the current `HEAD` overlay after pull.
- Do not suppress the follow-up repository state event for pull-based sync refreshes; if the Git extension publishes fresher state immediately after the explicit refresh, the graph can correct itself.
- Make webview metadata patches fall back to a full state render when the visible node set changes, so removed nodes cannot remain in the DOM after sync.
- Preserve existing sync guards for inherited upstream tracking, conflicts, dirty worktrees, no-upstream state, and Git failures.

## Non-Goals

- Changing Git pull/push behavior.
- Changing sync labels, confirmations, or context-menu visibility.
- Removing metadata patch behavior for push-only sync.
- Publishing or bumping the extension version.

## Acceptance Criteria

- Push-only sync still schedules `metadata-patch`.
- Pull-only sync schedules `full-rebuild` without suppressing follow-up repository events.
- Diverged sync continues to schedule `full-rebuild` without suppressing follow-up repository events.
- Current `HEAD` is placed on the commit reported by fresh refs when `repository.state.HEAD.commit` lags after pull.
- Partial metadata patching is skipped when commits enter or leave the visible scene.
- Existing sync messages and Git API calls remain unchanged.
