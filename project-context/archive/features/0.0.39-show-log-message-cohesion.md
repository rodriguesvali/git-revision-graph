# Show Log Message Cohesion

## Goal

Continue the `0.0.39` architectural cohesion work by separating Show Log webview message validation and dispatch from the editor panel provider lifecycle.

## Scope

- Add `src/showLog/messageHandler.ts` as the typed owner of Show Log webview message dispatch.
- Keep `src/showLog/messageValidation.ts` as the message-shape validation boundary.
- Route `ShowLogViewProvider` message handling through the dispatcher while preserving the existing private action methods.
- Preserve all current Show Log message names, payload shapes, panel behavior, Git operations, and clipboard/external-open actions.
- Add focused tests for valid and malformed Show Log message dispatch.

## Acceptance Criteria

- Invalid Show Log webview messages are ignored before they reach panel actions.
- Ready, log-scope, filter, pagination, file action, commit action, compare, GitHub, and reset messages route to the same provider actions as before.
- The Show Log provider retains ownership of panel lifecycle, state updates, loading/cancellation, and VS Code workbench integrations.
- Focused Show Log message and webview validation tests pass.
- `npm run build` passes.
- `npm test` passes.
- `git diff --check` passes.

## Verification

- Focused Show Log message handler, webview message validation, and Show Log webview tests passed with 11 tests after extracting the dispatcher. This includes `npm run build` through the focused validation command.
- `npm test` passed with 321 tests after extracting Show Log message dispatch. This includes `npm run build` through the test script.
- `git diff --check` passed after extracting Show Log message dispatch and updating verification artifacts.

## Risks

- Message routing could drift if a new Show Log message is added to validation without a dispatcher handler.
- The provider still owns the Show Log loading workflow; a future slice may split loading/cancellation if further architectural cohesion is needed.
- Runtime integration still needs manual Extension Development Host smoke testing because message delivery depends on a live VS Code webview.
