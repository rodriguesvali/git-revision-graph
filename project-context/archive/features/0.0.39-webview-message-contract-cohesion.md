# Webview Message Contract Cohesion

## Goal

Continue the `0.0.39` architectural cohesion work by centralizing revision graph host/webview message construction before any future message-name or payload changes.

## Scope

- Add `src/revisionGraph/hostMessages.ts` as the typed owner of host message builders for revision graph state, loading, error, and remote tag publication messages.
- Add `src/revisionGraph/webview/script/messages.ts` as the generated webview runtime owner of webview-to-host message builders.
- Route `RevisionGraphController` host message creation through the shared builders.
- Route `RevisionGraphViewStateWorkflow` cancel-state reposting through the shared update-state builder.
- Route revision graph toolbar, trace, and context-menu webview action messages through the generated runtime builders.
- Preserve the existing host/webview message names and payload shapes.
- Add focused tests that lock the builder payloads for `init-state`, `update-state`, `set-loading`, `set-error`, and `set-remote-tag-state`.

## Acceptance Criteria

- Existing `RevisionGraphViewHostMessage` union remains compatible with current webview runtime handling.
- `init-state` and `update-state` messages keep the current state payload shape.
- Loading, error, and remote tag publication messages keep their current payload shapes.
- Repository-pick cancel behavior still reposts the current state without scheduling a refresh.
- The generated revision graph webview still handles current host message names.
- The generated revision graph webview still posts current action message names.
- Focused host message, view-state workflow, and webview tests pass.
- `npm run build` passes.
- `npm test` passes.
- `git diff --check` passes.

## Verification

- `npm test` passed with 318 tests after adding host message builders. This includes `npm run build` through the test script.
- `git diff --check` passed after adding host message builders and updating verification artifacts.
- Focused revision graph webview and webview message validation tests passed with 30 tests after adding webview-to-host message builders.
- `npm test` passed with 318 tests after adding webview-to-host message builders. This includes `npm run build` through the test script.
- `git diff --check` passed after adding webview-to-host message builders and updating verification artifacts.

## Risks

- Host/webview compatibility can drift if a builder changes a message name or omits a payload field.
- Webview action routing can drift if generated builders are rendered after bootstrap or interaction handlers.
- Regex-heavy webview tests can miss runtime delivery behavior, so manual Extension Development Host smoke testing remains needed before release.
