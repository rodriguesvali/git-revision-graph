# 1.0.1 Pull Error Loading Clear Hotfix

Build implementation complete for the `1.0.1` current-branch pull error loading hotfix.

## Goal

Clear the revision graph `Pulling current branch...` loading feedback promptly when the current-branch pull operation fails, even while the VS Code error message remains visible.

## Context

The graph toolbar `Pull` action shows local webview loading feedback before sending `pull-current-head` to the extension host. If the Git pull fails, `pullCurrentBranchFromUpstream` previously awaited `showErrorMessage(...)` before returning `false`. The current-head workflow only reposts the current graph state after the action returns, so the webview loading overlay stayed visible until the user closed the error message.

## Scope

- Let pull failure error messages display without blocking the pull action return.
- Preserve prepared-refresh cancellation on pull failure.
- Preserve Source Control handoff for pull failures that leave conflicts to resolve, but keep that handoff sequenced after the error message closes.
- Add regression coverage that keeps a simulated error message open and verifies the pull action still returns `false`.

## Non-Goals

- No change to successful pull refresh behavior.
- No change to pull preconditions, clean-worktree guards, upstream checks, or conflict detection.
- No broad redesign of webview loading helpers or VS Code notification handling.
- No VSIX packaging or Marketplace publication without explicit maintainer approval.

## Acceptance Criteria

- When pull fails, the action returns `false` immediately after showing the error message.
- The current-head workflow can repost current state and clear webview loading while the error message remains visible.
- Prepared refresh suppression is canceled on pull failure.
- Source Control still opens after the error message closes when a pull failure should reveal conflicts.
- `npm run build`, focused ref action tests, `npm test`, and `git diff --check` pass.

## Risks

- Error-message handling must not swallow Source Control handoff for conflict-producing pull failures.
- Returning before the user closes the error message changes notification sequencing, so the change is intentionally limited to the pull failure path that controls this loading feedback.
