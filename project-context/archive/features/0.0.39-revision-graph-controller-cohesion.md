# Revision Graph Controller Cohesion

## Goal

Start the `0.0.39` controller cohesion work by extracting narrow responsibilities from `RevisionGraphController` without changing graph behavior, command IDs, message names, view types, or Git workflows.

## Scope

- Add a revision graph message dispatcher that owns webview payload validation and current-state/current-repository gating before controller action handling.
- Move the validated revision graph action routing out of `RevisionGraphController` into a focused message handler with injected host boundaries for VS Code clipboard, repository picking, unified diff, refresh, and state posting.
- Consolidate repository-required message actions behind a shared current-repository guard inside `RevisionGraphMessageHandler`.
- Move common graph reference actions such as compare, checkout, branch/tag creation, publish, reset, delete, merge, and abort merge into a focused `RevisionGraphRefActionWorkflow`.
- Move repository picking, canceled-pick state reposting, projection option normalization, and projection refresh scheduling into a focused `RevisionGraphViewStateWorkflow`.
- Move remote tag publication resolution, push-state updates, and delete-state updates into a focused `RevisionGraphRemoteTagWorkflow`.
- Move current `HEAD` sync, pull, push, and current-state repost handling into a focused `RevisionGraphCurrentHeadWorkflow`.
- Move revision graph fetch execution, fetch option picking, Git API/CLI selection, loading feedback, and refresh scheduling into a focused fetch workflow helper.
- Move load tracing output-channel ownership and host trace context injection into a focused trace service.
- Keep repository lifecycle behavior in the controller while narrowing controller ownership to view lifecycle, repository selection, render coordination, and host wiring.

## Acceptance Criteria

- Host/webview message names and payloads remain unchanged.
- Fetch still supports Git API fetch for normal options and Git CLI fetch when tags are requested.
- Load tracing still writes the same output-channel event format when `gitRevisionGraph.traceLoading` is enabled.
- Repository-scoped messages are still rejected while loading or when stale for the current repository.
- Validated graph actions still route to the same ref workflows, webview state posts, refresh requests, Show Log, copy, diff, fetch, and remote tag publication flows.
- Repository-required handler actions still no-op when no current repository is available.
- Common reference action targets still preserve the graph-provided ref names, labels, kinds, and reset options.
- Repository picking still reposts the current state when canceled and schedules a full rebuild when a repository is selected.
- Projection option changes still normalize current-branch descendant behavior and schedule a full rebuild.
- Remote tag state resolution still reports `unknown` when remote lookup cannot be completed.
- Current `HEAD` sync, pull, and push still repost the current graph state when no refresh is scheduled.
- `npm run build` passes.
- `npm test` passes.

## Risks

- Moving message gating can accidentally allow stale webview actions if repository-path checks drift.
- Moving the action switch can drift host side effects if injected controller boundaries do not preserve the original callback timing.
- Moving common reference actions can drift target labels or ref kinds if the message-to-target mapping changes.
- Moving view state actions can drift canceled repository-pick feedback or projection normalization if the refresh path changes.
- Moving remote tag publication handling can drift stale-response guards if request contexts are not created before async remote lookup or mutation.
- Moving current `HEAD` actions can drift graph state feedback if the no-refresh path stops reposting current state.
- Moving fetch can alter cancellation or refresh timing if the controller refresh request is not preserved.
- Moving tracing can lose trace context if request IDs are no longer read from the active render coordinator.
