# 0.0.24 Minimap Sync Performance

## Goal
Reduce minimap rendering cost during frequent viewport changes in large revision graphs.

## User Workflow
Users scroll, drag, zoom, and navigate through large graphs while the minimap remains responsive.

## Scope
- Batch minimap synchronization with `requestAnimationFrame`.
- Keep full minimap SVG rebuilds for layout, graph, and minimap zoom changes.
- Use a viewport-only fast path for graph viewport scroll and viewport restoration.
- Preserve existing minimap controls, placement, SVG structure, and navigation behavior.

## Non-Goals
- Changing the graph layout algorithm.
- Caching ELK layout output.
- Redesigning minimap visuals or controls.
- Adding settings.

## Touched Surfaces
- `src/revisionGraph/webview/script/bootstrap.ts`
- `src/revisionGraph/webview/script/layout.ts`
- `test/revisionGraphWebview.test.ts`

## Acceptance Criteria
- Repeated minimap sync requests are coalesced into a single animation frame.
- Scroll-triggered minimap updates avoid rebuilding minimap nodes and edges.
- Full minimap rebuilds still occur when graph content, node positions, or minimap zoom can change.
- Existing minimap navigation and viewport visibility behavior remain intact.

## Test Plan
- Run `npm run build`.
- Run `npm test`.

## Manual Validation
- Open a large revision graph in the Extension Development Host.
- Scroll and drag the main graph while watching the minimap viewport update smoothly.
- Use minimap zoom controls and verify mini nodes/edges still redraw at the new scale.
- Reorganize the graph and verify the minimap content updates to the new node positions.

## Release-Note Impact
- Improved minimap performance by batching updates and avoiding full SVG rebuilds during viewport-only movement.

## Handoff Notes
- This keeps performance work local to the webview runtime. A later cycle can add lightweight timing instrumentation if real-world traces are needed.
